# MemoChat 数据库架构

> 当前版本：2026-04-26
> 当前原则：本地数据库全部运行在 Docker；当前架构不使用 MySQL。

## 1. 总览

MemoChat 使用多存储架构，每种存储承担不同的数据一致性和性能职责：

| 存储 | 当前容器 | 宿主机端口 | 职责 | 是否最终事实来源 |
|------|----------|------------|------|------------------|
| PostgreSQL 16 | `memochat-postgres` | `15432` | 用户、关系、群组、会话索引、消息索引、运营数据、AI 元数据 | 是 |
| MongoDB 7 | `memochat-mongo` | `27017` | 私聊/群聊消息正文、动态内容、扩展文档 | 是 |
| Redis 7 | `memochat-redis` | `6379` | 验证码、会话缓存、在线状态、路由、短 TTL 缓存、锁 | 否 |
| MinIO | `memochat-minio` | `9000`, `9001` | 头像、图片、文件、视频等对象存储 | 是，文件事实来源 |
| Qdrant | `memochat-qdrant` | `6333`, `6334` | AI/RAG 向量检索 | 可重建 |
| Neo4j | `memochat-neo4j` | `7474`, `7687` | AI 图谱记忆、关系图谱 | 可由业务数据重建或半事实 |
| InfluxDB | `memochat-influxdb` | `8086` | 长期指标 | 观测数据 |

本地 Postgres 连接端口是 `15432`。容器内部和 Kubernetes Service 内部可以仍是 `5432`。

## 2. 为什么不用 MySQL

当前架构已经完成向 PostgreSQL 的迁移：

- 关系型主库统一为 PostgreSQL。
- MySQL 不再作为本地依赖启动。
- Docker Compose 不再包含 MySQL 服务。
- 旧 MySQL 数据只作为历史迁移来源，不能作为当前运行依赖。

因此，新的表设计、DAO、脚本、文档和部署说明都不应再新增 MySQL 入口。

## 3. PostgreSQL

### 3.1 连接配置

```ini
[Postgres]
Host=127.0.0.1
Port=15432
User=memochat
Passwd=123456
Database=memo_pg
Schema=memo
SslMode=disable
```

容器检查：

```powershell
docker exec memochat-postgres psql -U memochat -d memo_pg -c "select 1;"
```

### 3.2 Schema 分层

| Schema | 职责 |
|--------|------|
| `memo` | 核心业务：用户、好友、群组、消息索引、媒体索引 |
| `memo_ops` | 运营平台：服务快照、日志索引、压测结果、告警规则 |
| AI 相关表 | 可在 `memo` 或独立 schema 中承载会话、文档、工具调用元数据 |

迁移脚本位置：

```text
apps/server/migrations/postgresql/business
apps/server/migrations/postgresql/memo_ops
infra/deploy/local/init/postgresql
```

### 3.3 核心业务表

| 表 | 职责 |
|----|------|
| `memo.user` | 用户账号、邮箱、昵称、头像、密码摘要等 |
| `memo.user_id` | 用户 ID 分配和映射 |
| `memo.friend` | 好友关系 |
| `memo.friend_apply` | 好友申请 |
| `memo.friend_tag` / `friend_apply_tag` | 好友标签和申请标签 |
| `memo.chat_group` | 群组元数据 |
| `memo.chat_group_member` | 群成员 |
| `memo.chat_group_apply` | 入群申请 |
| `memo.chat_group_admin_permission` | 群管理权限 |
| `memo.chat_dialog` | 会话列表 |
| `memo.chat_dialog_meta` | 会话置顶、免打扰、草稿等扩展 |
| `memo.chat_private_msg` | 私聊消息索引 |
| `memo.chat_group_msg` | 群聊消息索引 |
| `memo.chat_private_read_state` | 私聊已读状态 |
| `memo.chat_group_read_state` | 群聊已读状态 |
| `memo.chat_media_asset` | 媒体对象元数据 |
| `memo.chat_call_session` | 通话会话元数据 |

消息正文不直接塞进 PostgreSQL 主表，而是写 MongoDB。PostgreSQL 保存可查询、可幂等、可关联的索引和元数据。

### 3.4 查询原则

用户、关系、群组和权限：

- 以 PostgreSQL 为准。
- Redis 只做短 TTL 缓存。
- 资料变更后应清理或刷新 Redis 缓存。

消息历史：

- PostgreSQL 查索引、分页、会话和未读。
- MongoDB 查消息正文。
- 大文件从 MinIO 读取。

一致性：

- 先写最终事实来源，再发异步事件。
- 使用 `msg_id`、唯一约束和 outbox 降低重复写入影响。
- 跨 PostgreSQL/MongoDB 的严格分布式事务不作为当前前提，依赖幂等、补偿和对账。

## 4. MongoDB

### 4.1 连接配置

```ini
[Mongo]
Enabled=true
Uri=mongodb://memochat_app:123456@localhost:27017/memochat
Database=memochat
```

检查：

```powershell
docker exec memochat-mongo mongosh "mongodb://memochat_app:123456@127.0.0.1:27017/memochat" --quiet --eval "db.getCollectionNames()"
```

### 4.2 集合

| 集合 | 职责 |
|------|------|
| `private_messages` | 私聊消息正文 |
| `group_messages` | 群聊消息正文 |
| `moments_content` | 动态/朋友圈内容 |

### 4.3 消息文档建议

```json
{
  "_id": "msg_id",
  "msg_id": "snowflake-or-uuid",
  "conversation_id": "private:uid_a:uid_b",
  "from_uid": 1001,
  "to_uid": 1002,
  "group_id": null,
  "content_type": "text|image|file|voice|video|system",
  "content": {},
  "media_assets": [],
  "client_time_ms": 1770000000000,
  "server_time_ms": 1770000000123,
  "trace_id": "..."
}
```

索引建议：

- `msg_id` 唯一。
- `conversation_id + server_time_ms`。
- `from_uid + server_time_ms`。
- 群聊按 `group_id + server_time_ms`。

## 5. Redis

### 5.1 连接配置

```ini
[Redis]
Host=127.0.0.1
Port=6379
Passwd=123456
```

检查：

```powershell
docker exec memochat-redis redis-cli -a 123456 ping
```

### 5.2 Key 类型

| Key 示例 | 类型 | 说明 |
|----------|------|------|
| `verify_code:{email}` | string | 邮箱验证码，TTL |
| `login_token:{uid}` | string/hash | 登录 token 或 session 信息 |
| `user:{uid}` | string/json | 用户资料缓存 |
| `online:{uid}` | string | 用户在线标记，TTL |
| `user_ip:{uid}` / `chatserver:{uid}` | string | 用户所在 ChatServer |
| `session:{uid}:{device}` | string | 设备会话 |
| `chatservers:{name}:online_users` | set | 某 ChatServer 当前在线用户集合 |
| `dialogs:{uid}` | zset | 对话列表热缓存 |
| `lock:{name}` | string | 分布式锁 |
| `rate:{scope}:{key}` | string | 限流计数 |

### 5.3 Redis 设计原则

- Redis 数据必须允许丢失。
- 在线状态必须有 TTL 或服务端 reconcile。
- 投递时如果 Redis 路由指向错误节点，要清理 stale route 并降级离线。
- 缓存更新优先采用“写 DB -> 删除缓存”而不是直接覆盖缓存。
- 验证码、限流、锁都必须设置 TTL。

## 6. MinIO

### 6.1 配置

```ini
[MinIO]
Enabled=true
Endpoint=127.0.0.1:9000
Region=us-east-1
AccessKey=memochat_admin
SecretKey=MinioPass2026!
BucketAvatar=memochat-avatar
BucketFile=memochat-file
BucketImage=memochat-image
BucketVideo=memochat-video
PublicUrl=http://127.0.0.1:9000
```

MinIO 保存对象本体，PostgreSQL/MongoDB 保存对象元数据、引用关系和消息关联。

### 6.2 典型对象

| Bucket | 内容 |
|--------|------|
| `memochat-avatar` | 头像 |
| `memochat-image` | 聊天图片 |
| `memochat-file` | 文件 |
| `memochat-video` | 视频 |

## 7. Qdrant 和 Neo4j

### 7.1 Qdrant

Qdrant 用于 AI/RAG 向量检索。业务数据库中保存文档、会话、权限和引用元数据；向量本身写 Qdrant。

检查：

```powershell
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:6333/collections
```

### 7.2 Neo4j

Neo4j 用于图谱记忆和关系分析。它可以保存用户上下文、实体、关系和长期记忆。

检查：

```powershell
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:7474
```

AI 图谱不应直接影响登录和聊天主链路；AI 组件失败时应降级，不应导致 IM 主功能不可用。

## 8. Redpanda 和 RabbitMQ 与数据库的关系

| 组件 | 数据性质 | 和数据库关系 |
|------|----------|--------------|
| Redpanda/Kafka | 事件流 | DB 写入后的异步边效应、审计、扩展消费 |
| RabbitMQ | 可靠任务 | 邮件、补投、重试、死信、outbox 补偿 |

消息发送时不要把 MQ 当最终存储。最终事实在 PostgreSQL/MongoDB/MinIO；MQ 负责异步和解耦。

## 9. 备份

备份目录建议：

```text
<repo-root>\backups
```

PostgreSQL：

```powershell
docker exec memochat-postgres pg_dump -U memochat -d memo_pg -Fc > <repo-root>\backups\memo_pg.dump
```

MongoDB：

```powershell
docker exec memochat-mongo mongodump --archive --gzip --uri "mongodb://memochat_app:123456@127.0.0.1:27017/memochat" > <repo-root>\backups\memochat_mongo.archive.gz
```

Redis：

```powershell
docker exec memochat-redis redis-cli -a 123456 SAVE
docker cp memochat-redis:/data/dump.rdb <repo-root>\backups\redis_dump.rdb
```

MinIO/Qdrant/Neo4j/InfluxDB 需要按数据目录或官方工具分别备份。备份前优先停写或做一致性快照。

## 10. 恢复原则

恢复优先级：

1. PostgreSQL：账号、关系、群组、消息索引先恢复。
2. MongoDB：消息正文恢复。
3. MinIO：媒体对象恢复。
4. Redis：可不恢复，允许由登录、心跳和查询重建。
5. Qdrant/Neo4j：AI 能力可重建时优先重建，不阻断主链路。

恢复后必须检查：

```powershell
docker exec memochat-postgres psql -U memochat -d memo_pg -c "select count(*) from memo.\"user\";"
docker exec memochat-mongo mongosh "mongodb://memochat_app:123456@127.0.0.1:27017/memochat" --quiet --eval "db.getCollectionNames().forEach(c => print(c + '=' + db.getCollection(c).countDocuments({})))"
docker exec memochat-redis redis-cli -a 123456 DBSIZE
```

## 12. 排查清单

### 12.1 确认没有本机数据库

```powershell
netstat -ano | findstr /R /C:":5432 .*LISTENING" /C:":3306 .*LISTENING" /C:":15432 .*LISTENING" /C:":27017 .*LISTENING" /C:":6379 .*LISTENING"
```

期望：

- 无 `5432` 和 `3306` 本机数据库监听。
- `15432`、`27017`、`6379` 由 Docker Desktop 后端占用。

### 12.2 WSL 误判

```powershell
wsl -d Ubuntu-24.04 -- bash -lc "ps -ef | grep -Ei 'postgres|mongod|redis-server|mysqld|qdrant|neo4j|redpanda|influx|rabbitmq|ollama' | grep -v grep || true"
wsl -d Ubuntu-24.04 -- bash -lc "dpkg -l | grep -Ei 'postgres|mongo|redis|mysql|qdrant|neo4j|redpanda|influx|rabbitmq|ollama' || true"
```

如果 WSL 里没有进程、包和 systemd 服务，只是 `ss` 能看到端口，那通常是 Docker Desktop 的转发。

### 12.3 登录数据检查

```powershell
docker exec memochat-postgres psql -U memochat -d memo_pg -c "select uid, email, name from memo.\"user\" order by uid desc limit 10;"
```

如果账号“不存在”，优先确认连接的是 Docker Postgres `15432`，而不是旧的本机/WSL 数据库。
