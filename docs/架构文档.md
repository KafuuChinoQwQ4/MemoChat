# MemoChat 架构文档

> 当前版本：2026-04-26
> 根目录：`<repo-root>`

MemoChat 是一个桌面 IM + AI 助手 + 本地运营平台项目。当前架构以 Qt/QML 客户端和 C++ 后端服务为主体，本地依赖全部运行在 Docker 中，业务进程在 Windows 上运行以便调试和迭代。

## 1. 架构目标

MemoChat 的核心目标：

- 支持账号注册、登录、验证码、好友关系、私聊、群聊、离线消息和历史消息。
- 支持媒体上传/下载、头像、图片、文件、语音/视频通话入口。
- 支持 TCP 和 QUIC 聊天长连接，HTTP 用于登录注册、资料、媒体和辅助 API。
- 支持 Redpanda/Kafka 事件流、RabbitMQ 任务流、Redis 在线路由和缓存。
- 支持 AI 助手、RAG、图谱记忆和本地模型调用。
- 支持本地可观测性：结构化日志、Zipkin/OTel、Prometheus、Grafana、Loki、Tempo、InfluxDB。
- 保持主工程和 R18 漫画源能力低耦合，R18 通过独立分区、路由和源插件协议扩展。

## 5. 客户端架构

客户端位于 `apps/client/desktop`。

### 5.1 `MemoChat-qml`

`MemoChat-qml` 是用户可见的 QML 应用。主要模块：

| 模块 | 职责 |
|------|------|
| `AuthController` | 登录、注册、验证码、重置密码 |
| `AppController` | 应用状态总控、页面切换、登录态协调 |
| `ChatController` | 私聊、群聊、消息发送、消息接收 |
| `ContactController` | 好友搜索、申请、验证、好友资料 |
| `ProfileController` | 用户资料编辑、头像更新 |
| `CallController` | 通话邀请、会话状态、LiveKit 桥接 |
| `MomentsController` | 动态/朋友圈 |
| `R18Controller` | R18 分区入口、漫画源/列表/详情交互 |
| `AgentController` | AI 助手 UI 和消息模型 |

QML 页面按领域放在：

```text
apps/client/desktop/MemoChat-qml/qml
```

R18 分区相关 UI 位于：

```text
apps/client/desktop/MemoChat-qml/qml/r18
```

### 5.2 `MemoChatShared`

共享库包含传输和基础状态：

| 文件/模块 | 职责 |
|-----------|------|
| `httpmgr.*` | HTTP 请求封装 |
| `tcpmgr.*` | TCP 长连接 |
| `QuicChatTransport.*` | QUIC 传输 |
| `IChatTransport.h` | 聊天传输抽象 |
| `ChatMessageDispatcher.*` | 消息分发 |
| `usermgr.*` / `userdata.*` | 用户态和用户数据 |
| `TelemetryUtils.*` | 客户端遥测工具 |

客户端登录后，GateServer 返回目标 ChatServer、transport 偏好和 `login_ticket`。客户端再通过 TCP 或 QUIC 连接 ChatServer，并使用 ticket 完成 ChatServer 登录。

## 6. 网关架构

### 6.1 GateServer

`apps/server/core/GateServer` 是当前主网关，职责包括：

- HTTP API：登录、注册、验证码、资料、媒体、动态、AI 路由入口。
- 认证：读取 PostgreSQL 用户数据，生成 token 和 `login_ticket`。
- ChatServer 选择：调用 StatusServer 获取目标节点。
- 媒体：通过 MinIO/S3 兼容接口上传/下载。
- 缓存：使用 Redis 缓存用户资料、登录状态和短期数据。
- 可观测性：结构化日志、trace id、Zipkin/OTel 输出。

关键依赖：

```ini
[Postgres] Port=15432
[Redis] Port=6379
[Mongo] Uri=mongodb://memochat_app:123456@localhost:27017/memochat
[MinIO] Endpoint=127.0.0.1:9000
[Kafka] Brokers=127.0.0.1:19092
[RabbitMQ] Host=127.0.0.1
```

### 6.2 GateServerCore

`GateServerCore` 抽出了可被 HTTP/1.1、HTTP/2、HTTP/3 目标复用的能力：

- `AuthLoginSupport`
- `MediaStorage` / `S3MediaStorage`
- `PostgresDao`
- `MongoDao`
- `RedisPipelines`
- gRPC clients
- Call/Moments/Profile 相关支持

### 6.3 HTTP/1.1、HTTP/2、HTTP/3 拆分目标

| 目录 | 定位 |
|------|------|
| `GateServerHttp1.1` | 独立 HTTP/1.1 目标，便于路由拆分和 R18 插件协议验证 |
| `GateServerHttp2` | 独立 HTTP/2 目标，基于 nghttp2/OpenSSL 能力 |
| `GateServerHttp3` | 独立 HTTP/3 目标，依赖 msquic/ngtcp2/nghttp3/OpenSSL 构建能力 |

这些目标不等价于主网关必须同时在线。当前本地登录和普通 API 仍以 `GateServer.exe :8080` 为主。

## 7. ChatServer 架构

`apps/server/core/ChatServer` 是 IM 核心。一个本地环境通常启动 4 个实例：

| 实例 | TCP | QUIC | gRPC |
|------|-----|------|------|
| chatserver1 | `8090` | `8190` | `50055` |
| chatserver2 | `8091` | `8191` | `50056` |
| chatserver3 | `8092` | `8192` | `50057` |
| chatserver4 | `8093` | `8193` | `50058` |

核心模块：

| 模块 | 职责 |
|------|------|
| `CServer` / `CSession` | TCP session 生命周期、帧解析、心跳 |
| `QuicChatServer` | QUIC listener 和 stream 处理 |
| `ChatSessionService` | ChatServer 登录、ticket 校验、会话绑定 |
| `PrivateMessageService` | 私聊发送、跨节点投递、离线处理 |
| `GroupMessageService` | 群聊发送、成员扩散、群消息持久化 |
| `ChatRelationService` | 好友、申请、关系 bootstrap |
| `MessageDeliveryService` | 投递路由、跨节点转发 |
| `ChatOutboxService` | outbox、重试和补偿 |
| `PostgresDao` | 用户、关系、群组、消息索引 |
| `MongoDao` | 消息正文 |
| `RedisMgr` | 在线状态、缓存、Pub/Sub |
| `KafkaAsyncEventBus` | Redpanda/Kafka 事件流 |
| `RabbitMQTaskBus` / task bus | 可靠任务分发 |

ChatServer 把 Redis 当作临时状态层，不把 Redis 当作最终事实来源。投递时如果 Redis 路由失效，必须清理 stale route 并允许降级/离线。

## 8. StatusServer

`StatusServer` 负责：

- 根据负载选择 ChatServer。
- 维护 ChatServer 节点配置。
- 读取 Redis 中的在线状态和节点状态。
- 对 GateServer 提供 gRPC 查询。
- 对 ChatServer 提供状态辅助。

本地端口：

```text
50052
```

## 9. VarifyServer

`VarifyServer` 当前是 C++ 服务目标，职责：

- 生成验证码。
- 写入 Redis 并设置 TTL。
- 发送邮件。
- 做邮箱/IP 限流。
- 通过 RabbitMQ 进行验证码邮件异步投递、重试和死信。

本地端口：

```text
gRPC: 50051
Health: 8081
```

## 10. AI 架构

AI 能力分两层：

| 层 | 服务 | 职责 |
|----|------|------|
| C++ 集成层 | `AIServer` | 与主服务和客户端协议集成 |
| Python 编排层 | `AIOrchestrator` | LLM、RAG、工具、图谱、MCP |

AIOrchestrator 在 Docker 中运行，连接：

- `memochat-ollama:11434`
- `memochat-qdrant:6333`
- `memochat-neo4j:7687`
- `host.docker.internal:15432` 的 Postgres

AI 数据边界：

| 数据 | 存储 |
|------|------|
| 会话元数据 | PostgreSQL |
| 文档向量 | Qdrant |
| 图谱记忆 | Neo4j |
| 模型推理 | Ollama |

## 11. R18 分区架构

R18 分区按低耦合方式集成：

```text
QML R18 页面
   │
R18Controller / R18ListModel
   │ HTTP
GateServerHttp1.1 / H1R18Routes
   │
R18PluginHost
   │
用户导入的漫画源定义 / C++ 源插件协议
```

设计原则：

- R18 分区只影响 R18 页面和 R18 路由。
- 不污染 ChatServer、登录、好友、群组、普通媒体和 AI 链路。
- 源导入使用统一协议描述搜索、详情、章节和图片解析。
- 可借鉴 Venera 的“直接拉取用户选定漫画源脚本、源协议抽象、主应用隔离”思想。当前客户端可以直接拉取 `jm.js` 等 JavaScript 源，也可以从 Venera 源目录选择源，并在服务端暂存为 `venera-js`；执行这些源还需要单独的 Venera JS API 运行时适配层，不把第三方爬虫逻辑硬编码进主业务。

## 12. 数据架构

| 数据类型 | 主存储 | 辅助层 |
|----------|--------|--------|
| 用户、好友、群组、申请、会话索引 | PostgreSQL | Redis 缓存 |
| 私聊/群聊消息正文 | MongoDB | PostgreSQL 索引 |
| 在线状态、验证码、短 TTL 缓存 | Redis | PostgreSQL/MongoDB 可重建 |
| 文件、头像、图片、视频 | MinIO | PostgreSQL/MongoDB 元数据 |
| 事件流 | Redpanda/Kafka | PostgreSQL outbox |
| 可靠任务 | RabbitMQ | DLQ/重试队列 |
| AI 向量 | Qdrant | PostgreSQL 元数据 |
| AI 图谱 | Neo4j | 原始业务数据 |

PostgreSQL 本地宿主机端口是 `15432`，不再使用本机 `5432`。MySQL 已退出当前架构。

## 13. 消息发送链路

私聊简化流程：

```text
Client A
  │ TCP/QUIC frame
  ▼
ChatServer A
  │ 校验 session/ticket/关系
  ├─ 写 PostgreSQL 消息索引
  ├─ 写 MongoDB 消息正文
  ├─ 更新 Redis 对话/在线路由
  ├─ 发布 Redpanda 事件
  ├─ 必要时投递 RabbitMQ 任务
  │
  ├─ 如果 B 在线且在本节点：直接推送
  ├─ 如果 B 在线但在其他节点：gRPC 转发到目标 ChatServer
  └─ 如果 B 离线：保持离线状态，登录 bootstrap 拉取
```

幂等依赖：

- `msg_id`
- PostgreSQL 唯一约束或去重逻辑
- MongoDB `_id`/消息 id
- outbox 状态

## 14. 协议

| 通道 | 用途 |
|------|------|
| HTTP/JSON | 登录、注册、验证码、资料、媒体、动态、AI 辅助 API |
| TCP binary frame | 主聊天长连接 |
| QUIC stream | 可选聊天长连接 |
| gRPC/Protobuf | 服务间调用 |
| Kafka protocol | Redpanda 事件流 |
| AMQP | RabbitMQ 任务流 |

ChatServer TCP/QUIC 使用统一业务帧和协议号。客户端选择 QUIC 失败时回退 TCP。

## 18. 生产化关注点

当前本地架构已具备生产化雏形，但真正生产部署还需要补齐：

- 密钥管理：数据库密码、SMTP、MinIO、RabbitMQ、JWT/HMAC secret 不应写死。
- 数据库 HA：PostgreSQL/MongoDB/Redis 需要主从、备份、恢复演练。
- 对象存储：生产使用外部 S3 或高可用 MinIO。
- 消息系统：Redpanda/RabbitMQ 需要持久化、监控和告警。
- TLS：HTTP、gRPC、QUIC、管理后台都应启用可信证书。
- 容器镜像：固定 tag，不使用 `latest`。
- Kubernetes：资源限制、探针、HPA、PVC、NetworkPolicy、Secret、Ingress。
- 审计：关键操作、登录、资料变更、管理员操作需要审计日志。
