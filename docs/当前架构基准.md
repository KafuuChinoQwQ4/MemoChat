# MemoChat 当前架构基准

> 基准日期：2026-04-26  
> 适用范围：本地开发、联调、自测、文档修订  
> 根目录：`D:\MemoChat-Qml-Drogon`

这份文档是当前仓库的事实基准。其他长文档中保留的历史讨论、迁移分析或旧目录截图，如果和这里冲突，以本文为准。

## 1. 总体形态

MemoChat 当前采用“Windows 原生业务服务 + Docker 基础设施”的本地开发形态：

- C++/Qt 业务进程在 Windows 上直接运行，便于调试、断点、日志窗口和快速构建。
- PostgreSQL、MongoDB、Redis、MinIO、Redpanda、RabbitMQ、Ollama、Qdrant、Neo4j、Prometheus、Grafana、Loki、Tempo、InfluxDB、OTel Collector 等依赖都运行在 Docker。
- Docker 容器使用固定宿主机端口暴露给业务进程，业务配置里使用 `127.0.0.1:<端口>`。
- WSL 不承载项目数据库或基础设施服务。WSL 中看到的相关监听通常是 Docker Desktop 的端口转发，不代表 WSL 内独立启动服务。

## 3. 服务职责

| 服务 | 运行形态 | 主要职责 | 主要依赖 |
|------|----------|----------|----------|
| `MemoChatQml` | Windows Qt/QML | 登录注册、聊天、联系人、群组、媒体、通话、R18 分区入口、AI 助手入口 | GateServer、ChatServer、LiveKit |
| `GateServer` | Windows C++ | HTTP API 网关、登录注册、验证码入口、媒体上传/下载、Moments、AI HTTP 路由、呼叫入口 | Postgres、Redis、Mongo、MinIO、Status/Varify gRPC |
| `GateServerHttp1.1` | Windows C++ 可选目标 | 独立 HTTP/1.1 路由实验和 R18 源插件接口验证 | GateServerCore、R18 插件协议 |
| `GateServerHttp2` | Windows C++ 可选目标 | HTTP/2 独立目标，复用部分网关能力 | nghttp2、OpenSSL、GateServerCore |
| `GateServerHttp3` | Windows C++ 可选目标 | HTTP/3 独立目标 | msquic/ngtcp2/nghttp3、OpenSSL |
| `StatusServer` | Windows C++ | ChatServer 负载选择、节点状态、在线路由辅助、登录目标分配 | Redis、Postgres、Chat gRPC |
| `ChatServer` | Windows C++ 多实例 | TCP/QUIC 长连接、消息收发、好友/群组业务、离线消息、历史拉取、跨节点投递 | Postgres、Mongo、Redis、Redpanda、RabbitMQ |
| `VarifyServer` | Windows C++ | 验证码生成、Redis 存储、邮件发送、限流、健康检查 | Redis、RabbitMQ、SMTP |
| `AIServer` | Windows C++ | AI 能力桥接、与 AIOrchestrator 协作 | AIOrchestrator |
| `AIOrchestrator` | Docker Python | LLM 编排、RAG、工具调用、图谱记忆、MCP 操作 | Ollama、Qdrant、Neo4j、Postgres |
| `MemoOps` | Windows/Python/QML | 本地运营和观测辅助 | 日志目录、Postgres/Influx/Grafana |

## 5. 当前连接配置

业务服务的本地配置以 `apps/server/core/*/config.ini` 和 `infra/Memo_ops/runtime/services/*/config.ini` 为准。

### 5.1 Postgres

```ini
[Postgres]
Host=127.0.0.1
Port=15432
User=memochat
Passwd=123456
Database=memo_pg
Schema=memo
SslMode=disable
```

说明：

- 宿主机端口 `15432` 是为避免和本机/WSL 上可能存在的 `5432` 冲突。
- 容器内部仍然是 PostgreSQL 标准端口 `5432`。
- Kubernetes 集群内服务发现可以继续使用 `5432`，因为那是 Pod/Service 内部端口，不是 Windows 宿主机端口。

### 5.2 MongoDB

```ini
[Mongo]
Enabled=true
Uri=mongodb://memochat_app:123456@localhost:27017/memochat
Database=memochat
```

当前集合分工：

- `private_messages`：私聊消息正文。
- `group_messages`：群聊消息正文。
- `moments_content`：动态/朋友圈内容。

### 5.3 Redis

```ini
[Redis]
Host=127.0.0.1
Port=6379
Passwd=123456
```

Redis 是临时状态和低延迟路由层，不是最终事实来源。在线路由、验证码、会话缓存、热点关系缓存都必须允许过期和重建。

### 5.4 Redpanda 和 RabbitMQ

```ini
[Kafka]
Brokers=127.0.0.1:19092

[RabbitMQ]
Host=127.0.0.1
Port=5672
Username=memochat
Password=123456
VHost=/
```

Redpanda 用于事件流和异步边效应，RabbitMQ 用于可 ACK/NACK、可重试、可死信的任务流。

## 7. 本地启动顺序

推荐顺序：

1. 启动 Docker Desktop。
2. 启动本地基础设施：

```powershell
docker compose -f infra\deploy\local\docker-compose.yml up -d
```

或只启动数据存储：

```powershell
docker compose -f infra\deploy\local\compose\datastores.yml up -d
```

3. 初始化 topic/topology/schema，如有需要：

```powershell
powershell -ExecutionPolicy Bypass -File tools\scripts\init_postgresql_schema.ps1
powershell -ExecutionPolicy Bypass -File tools\scripts\init_kafka_topics.ps1
powershell -ExecutionPolicy Bypass -File tools\scripts\init_rabbitmq_topology.ps1
```

4. 构建 C++/QML 产物。
5. 部署到运行时目录：

```powershell
tools\scripts\status\deploy_services.bat
```

6. 启动本地服务窗口：

```powershell
tools\scripts\status\start-all-services.bat
```

7. 访问客户端或直接调用接口自测。

### 8.3 登录自测

```powershell
curl.exe --% -sS -H "Content-Type: application/json" -d "{\"email\":\"1220743777@qq.com\",\"passwd\":\"745230\",\"client_ver\":\"3.0.0\"}" http://127.0.0.1:8080/user_login
```

成功响应应包含：

- `error: 0`

## 9. 数据职责边界

| 数据 | 最终事实来源 | 辅助存储 | 说明 |
|------|--------------|----------|------|
| 用户账号、密码、资料 | PostgreSQL | Redis 用户缓存 | 登录认证以 PostgreSQL 为准 |
| 好友关系、申请、标签 | PostgreSQL | Redis 短 TTL 缓存 | 关系变更必须落库 |
| 群组、成员、权限 | PostgreSQL | Redis 缓存 | 权限判断优先保证一致性 |
| 私聊/群聊消息索引 | PostgreSQL | Kafka/RabbitMQ outbox | 用于翻页、未读、幂等和审计 |
| 私聊/群聊消息正文 | MongoDB | MinIO 存大文件 | 文档模型便于扩展消息类型 |
| 在线状态、所在 ChatServer | Redis | ChatServer 内存 | 临时状态，可重建 |
| 验证码、限流计数 | Redis | PostgreSQL 审计可选 | TTL 数据 |
| 媒体文件 | MinIO | PostgreSQL/Mongo 元数据 | 本地 S3 兼容 |
| AI 向量 | Qdrant | PostgreSQL 元数据 | RAG 检索 |
| AI 图谱 | Neo4j | PostgreSQL/Mongo 原始数据 | 关系和记忆图谱 |

## 10. R18 漫画源扩展边界

当前 R18 集成采用“客户端 UI + 网关路由 + C++ 插件协议”的低耦合方式：

- 客户端入口在 `apps/client/desktop/MemoChat-qml/R18Controller.*`、`R18ListModel.*` 和 `qml/r18/`。
- 服务端 HTTP/1.1 拆分目标中包含 `H1R18Routes.*`、`R18PluginHost.*`、`R18PluginProtocol.h`。
- 漫画源导入应以用户可导入源为目标，源定义建议使用 JSON/YAML，运行时经插件协议转换为统一的搜索、详情、章节、图片接口。
- 不建议把某个第三方爬虫仓库的 Python 逻辑硬编码进主工程；优先保留源插件协议和主工程解耦。
- R18 能力应只挂在 R18 分区，不影响登录、聊天、媒体、AI 和普通动态链路。
