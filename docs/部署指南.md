# MemoChat 部署指南

> 当前版本：2026-04-26
> 根目录：`<repo-root>`

本文说明当前 MemoChat 的本地部署、Docker 依赖、Windows 原生服务启动、AI 组件和生产/Kubernetes 部署边界。旧版文档中出现的根目录 `deploy/`、`scripts/`、`server/`、`client/` 已迁移到 `infra/deploy/`、`tools/scripts/`、`apps/server/core/`、`apps/client/desktop/`。

## 1. 部署形态

MemoChat 支持三种部署形态：

| 形态 | 用途 | 业务服务 | 基础设施 |
|------|------|----------|----------|
| 本地开发 | 日常开发、调试、联调 | Windows 原生进程 | Docker 容器 |
| 本地 Compose | 快速拉起依赖或部分服务 | 可选容器化 | Docker Compose |
| Kubernetes | 生产/准生产演练 | Pod/Deployment | 集群内 StatefulSet/外部托管 |

当前最常用的是第一种：业务服务在 Windows 上跑，数据库和基础设施全部在 Docker 里跑。

## 2. 本地部署总览

### 2.1 基础设施必须 Docker-only

本地基础设施包括：

| 服务 | 容器名 | 宿主机端口 | 说明 |
|------|--------|------------|------|
| PostgreSQL | `memochat-postgres` | `15432` | 主关系数据库 |
| MongoDB | `memochat-mongo` | `27017` | 消息正文和动态内容 |
| Redis | `memochat-redis` | `6379` | 缓存、验证码、在线路由 |
| MinIO | `memochat-minio` | `9000`, `9001` | 本地 S3 兼容对象存储 |
| Redpanda | `memochat-redpanda` | `19092`, `18082` | Kafka 兼容事件总线 |
| RabbitMQ | `memochat-rabbitmq` | `5672`, `15672` | 任务队列、重试、死信 |
| Ollama | `memochat-ollama` | `11434` | 本地 LLM |
| Qdrant | `memochat-qdrant` | `6333`, `6334` | 向量检索 |
| Neo4j | `memochat-neo4j` | `7474`, `7687` | 图谱和记忆关系 |
| Prometheus | `memochat-prometheus` | `9090` | 指标 |
| InfluxDB | `memochat-influxdb` | `8086` | 长期指标 |
| Grafana | `memochat-grafana` | `3000` | 可视化 |
| Loki | `memochat-loki` | `3100` | 日志 |
| Tempo | `memochat-tempo` | `3200` | 链路追踪 |
| OTel Collector | `memochat-otel-collector` | `4317`, `4318`, `9411`, `9464` | 遥测收集 |

当前不使用 MySQL，不需要启动 `3306`。

### 2.2 一键启动本地依赖

```powershell
cd <repo-root>
docker compose -f infra\deploy\local\docker-compose.yml up -d
```

只启动数据库和缓存：

```powershell
docker compose -f infra\deploy\local\compose\datastores.yml up -d
```

单独启动消息队列和观测组件：

```powershell
docker compose -f infra\deploy\local\compose\kafka.yml up -d
docker compose -f infra\deploy\local\compose\rabbitmq.yml up -d
docker compose -f infra\deploy\local\compose\observability.yml up -d
```

### 2.3 容器自启动策略

核心容器应使用：

```text
restart: unless-stopped
```

验证：

```powershell
docker inspect $(docker ps -q) --format "{{.Name}} {{.HostConfig.RestartPolicy.Name}}"
```

期望输出中项目容器均为 `unless-stopped`。

## 3. 初始化

### 3.1 PostgreSQL

初始化入口：

```powershell
powershell -ExecutionPolicy Bypass -File tools\scripts\init_postgresql_schema.ps1
```

主要脚本位置：

```text
infra/deploy/local/init/postgresql
apps/server/migrations/postgresql/business
apps/server/migrations/postgresql/memo_ops
```

连接方式：

```powershell
docker exec -it memochat-postgres psql -U memochat -d memo_pg
```

宿主机连接参数：

```text
host=127.0.0.1
port=15432
user=memochat
password=123456
database=memo_pg
schema=memo
```

### 3.2 MongoDB

初始化脚本：

```text
infra/deploy/local/init/mongo/001-init.js
```

连接检查：

```powershell
docker exec memochat-mongo mongosh -u root -p 123456 --authenticationDatabase admin --quiet --eval "db.adminCommand({ ping: 1 })"
```

业务连接 URI：

```text
mongodb://memochat_app:123456@127.0.0.1:27017/memochat
```

### 3.3 Redis

连接检查：

```powershell
docker exec memochat-redis redis-cli -a 123456 ping
```

Redis 保存验证码、在线状态、用户所在 ChatServer、会话缓存、分布式锁和短 TTL 关系缓存。它不是最终事实来源，数据丢失后应能通过 PostgreSQL/MongoDB 重建。

### 3.4 Kafka/Redpanda

初始化 topic：

```powershell
powershell -ExecutionPolicy Bypass -File tools\scripts\init_kafka_topics.ps1
```

本地 broker：

```text
127.0.0.1:19092
```

常用 topic：

- `chat.private.v1`
- `chat.group.v1`
- `chat.private.dlq.v1`
- `chat.group.dlq.v1`
- `dialog.sync.v1`
- `relation.state.v1`
- `user.profile.changed.v1`
- `audit.login.v1`

### 3.5 RabbitMQ

初始化 topology：

```powershell
powershell -ExecutionPolicy Bypass -File tools\scripts\init_rabbitmq_topology.ps1
```

本地连接：

```text
AMQP: 127.0.0.1:5672
Management: http://127.0.0.1:15672
User: memochat
Password: 123456
```

RabbitMQ 用于需要可靠 ACK、重试、死信和延迟补偿的任务，不替代 Redpanda 的事件流职责。

## 4. 构建

### 4.1 全量构建

```powershell
cmake --preset msvc2022-all
cmake --build --preset msvc2022-all-release
```

### 4.2 按目标构建

```powershell
cmake -B build -S . -G Ninja -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release --target GateServer ChatServer StatusServer VarifyServer MemoChatQml
```

### 4.3 顶层构建选项

| 选项 | 说明 |
|------|------|
| `BUILD_SERVER` | 构建后端服务 |
| `BUILD_CLIENT` | 构建 QML 客户端 |
| `BUILD_OPS` | 构建 MemoOps |
| `BUILD_LOADTEST` | 构建压测工具 |
| `BUILD_TESTS` | 构建测试 |

## 5. 部署本地运行时目录

构建产物默认在：

```text
build/bin/Release
```

本地运行时目录在：

```text
infra/Memo_ops/runtime/services
```

部署命令：

```powershell
tools\scripts\status\deploy_services.bat
```

该脚本会复制 exe、配置和必要 dll。它不会启动 Docker 依赖，因此先确保 Docker 容器健康。

## 6. 启动 Windows 原生服务

```powershell
tools\scripts\status\start-all-services.bat
```

脚本会启动：

- `GateServer`
- `StatusServer`
- `ChatServer-1`
- `ChatServer-2`
- `ChatServer-3`
- `ChatServer-4`
- `VarifyServer`
- 可选客户端/运营平台，取决于脚本配置和部署结果

停止：

```powershell
tools\scripts\status\stop-all-services.bat
```

日志目录：

```text
infra/Memo_ops/artifacts/logs/services
```

每个服务窗口也会显示自己的 stdout/stderr。

## 8. AI 组件部署

AI 本地依赖：

| 组件 | 容器 | 端口 |
|------|------|------|
| AIOrchestrator | `memochat-ai-orchestrator` | `8096` |
| Ollama | `memochat-ollama` | `11434` |

| Neo4j | `memochat-neo4j` | `7474`, `7687` |

AIOrchestrator 在容器中连接 Postgres 时使用：

```text

```

## 10. Kubernetes 部署边界

Kubernetes 配置位于：

```text
infra/deploy/kubernetes
infra/helm
```

Kubernetes 中的数据库端口可以是集群内部标准端口，例如 PostgreSQL `5432`。这和本地 Windows 宿主机端口 `15432` 不冲突。

部署前要明确：

- 开发环境可以使用 Docker Desktop Kubernetes。
- 生产环境应使用托管数据库/存储或 StatefulSet，并配置备份、PVC、资源限制和安全密钥。
- 镜像构建文件位于 `infra/deploy/images`。
- 不要把本地开发密码直接用于生产环境。

## 11. 备份和恢复

备份建议写到：

```text
<repo-root>\backups
```

PostgreSQL：

```powershell
docker exec memochat-postgres pg_dump -U memochat -d memo_pg -Fc > <repo-root>\backups\memo_pg.dump
```

MongoDB：

```powershell
docker exec memochat-mongo mongodump --archive --gzip --uri "mongodb://memochat_app:123456@127.0.0.1:27017/memochat" > <repo-root>\backups\memochat_mongo.archive.gz
```

Redis：

```powershell
docker exec memochat-redis redis-cli -a 123456 SAVE
docker cp memochat-redis:/data/dump.rdb <repo-root>\backups\redis_dump.rdb
```

MinIO、Qdrant、Neo4j、InfluxDB 等也应按各自数据目录或官方工具进行备份。不要删除 `D:\docker-data\memochat` 下的数据目录，除非已经有可恢复备份。

## 12. 故障排查

### 12.1 Docker 容器没启动

```powershell
docker compose -f infra\deploy\local\docker-compose.yml ps
docker compose -f infra\deploy\local\docker-compose.yml logs --tail=100 <service>
```

### 12.2 端口冲突

```powershell
netstat -ano | findstr "<port>"
Get-Process -Id <pid>
```

数据库和基础设施端口应由 Docker Desktop 后端占用。`5432` 和 `3306` 不应被本机数据库占用。

### 12.3 服务窗口无输出

查看：

```text
infra/Memo_ops/artifacts/logs/services
```

以及对应窗口顶部显示的 exe、config、log 路径。

### 12.4 登录失败但数据库有数据

依次检查：

1. `GateServer` 是否健康。
2. `StatusServer` 是否监听 `50052`。
3. `ChatServer1-4` 是否监听 `8090-8093` 和 `50055-50058`。
4. `Postgres` 是否为 `15432`。
5. `Redis` 是否可写入登录 token 和路由。
6. `ChatAuth.HmacSecret` 在 GateServer 和 ChatServer 是否一致。

### 12.5 WSL 中看到 Docker 端口

这是 Docker Desktop 的 WSL 转发层。确认 WSL 没有独立服务：

```powershell
wsl -d Ubuntu-24.04 -- bash -lc "ps -ef | grep -Ei 'postgres|mongod|redis-server|qdrant|neo4j|redpanda|influx|rabbitmq|ollama' | grep -v grep || true"
wsl -d Ubuntu-24.04 -- bash -lc "systemctl list-units --type=service --all --no-pager | grep -Ei 'postgres|mongo|redis|qdrant|neo4j|redpanda|influx|rabbitmq|ollama' || true"
```

如果没有进程和 systemd 服务，就不是 WSL 自己在跑。
