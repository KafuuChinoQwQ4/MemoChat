MemoChat 项目架构概览

主链路

登录注册

客户端走 Envoy/GateServer 的 HTTP API；GateServer 处理账号、token、profile、media 等入口，验证码走 VarifyServer 与 Redis，账号数据落 Postgres，登录后拿到 chat 路由和 login_ticket。

聊天长连

客户端按登录结果连接 ChatServer 的 TCP/QUIC 通道；ChatServer 负责在线会话、心跳、消息分发，并把 message、relation、delivery、outbox 等工作拆到内部 domain/worker 边界。

AI/RAG

AI 请求可以经 GateServer 的 AI route proxy，也可以经 AIServer 这个 C++ bridge；最终进入 Python FastAPI 的 AIOrchestrator，处理 /chat、/smart、/kb、/agent、/pet 等能力。

媒体通话

头像、图片、视频、动态等对象存储走 MinIO/S3 抽象；通话控制在 GateServerCore 的 call 服务侧，运行时依赖 LiveKit/Coturn 这类可选实时音视频基础设施。

模块边界

Client

  • AppController 是 app shell，不承接 feature 业务循环。
  • features/* 自己闭合 controller、viewmodel、model、service/cache。

Gate

  • 认证、资料、媒体、call、moments、AI proxy 的 HTTP 聚合入口。
  • GateServerCore 复用账号、token、媒体、通话等共享逻辑。

Chat

  • 长连接入口和在线状态在 transport/session。
  • message、relation、delivery、persistence、messaging 已分层。

AI

  • AIOrchestrator 是 FastAPI 编排层。
  • RAG 依赖 Qdrant/Neo4j，任务与事件可接 RabbitMQ/Redpanda。
