## 2. 本次优化动作

- Gate 登录审计从主路径同步发送改为异步投递，取消每条登录请求上的 Kafka `flush()`。
- Python 压测器改为 `http.client` keep-alive + 双 Gate 轮询，减少短连接开销。
- ChatServer 热路径移除 `cout`/`cerr` 调试输出，避免控制台 I/O 挤占 CPU。
- 修复 `ID_SYNC_DRAFT_REQ (1072)` / `ID_PIN_DIALOG_REQ (1074)` 对 `group_id` 的无条件 variant 读取，解决 `bad variant access` 崩溃。
- 将服务日志级别调为 `warn`，并关闭控制台输出，降低压测噪声。
- Chat TCP 登录继续优化：删除重复 `USERIPPREFIX` Redis 写入、删除登录后 Redis 验证 `GET` 和高频 verify 日志。
- 新增压测 FeatureFlag：`chat_login_offline_push=false` 将离线消息补推移出登录压测主路径，`chat_login_verbose_logs=false` 降低每次登录的结构化日志开销。
- 新增唯一用户登录压测模式：`chat_login_duplicate_session_check=false`。该模式适用于 2000 个压测账号互不重复登录的场景，跳过同账号重复登录互踢锁和路由查询；业务默认不配置时仍保持开启。

## 6. 瓶颈结论

1. HTTP 登录主路径已从“同步审计 + 短连接”优化为“异步审计 + keep-alive + 双 Gate”，吞吐提升到约 `1240.9 QPS`，P95 下降到 `79.1ms`。
2. TCP 登录主要瓶颈已从离线补推线程、重复 Redis 往返和高频日志，转移到 Gate HTTP 登录、Redis 路由写入和本机多服务资源抢占。
3. 唯一用户登录压测模式下，50 并发端到端 TCP 登录达到 `371.054 QPS`、`P95=177.372ms`，相比旧基线吞吐提升约 `755.6%`。
4. Redis、MongoDB、Qdrant 本地吞吐远高于登录主链路，短期不是主瓶颈。
5. Postgres 短连接模型下延迟仍偏高，是尾延迟的重要来源之一。
6. RAG 质量可控，`recall@k = 100%`、`hallucination_rate = 0%`，但编排链路 QPS 仍偏低，适合作为 AI 侧继续优化的重点。
