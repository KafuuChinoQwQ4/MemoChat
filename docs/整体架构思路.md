## 一、项目整体架构总览

### 1.1 系统定位

MemoChat 是一个**企业级即时通讯（IM）平台**，采用微服务架构，支持 HTTP/1.1、HTTP/3（QUIC）、TCP 长连接和 gRPC 多协议客户端接入。项目分为两大部分：

- **客户端**：Qt/QML 桌面应用（Windows），支持私聊、群聊、朋友圈、音视频通话
- **服务端**：C++ 微服务集群（GateServer、StatusServer、ChatServer、VarifyServer），支持水平扩展

### 1.2 服务划分与通信拓扑

```
┌──────────────────────────────────────────────────────────────────────┐
│                          QML 客户端                                   │
│   HTTP 登录注册 │ TCP/QUIC 长连接聊天 │ WebRTC/LiveKit 音视频通话    │
└──────────────────────────────┬───────────────────────────────────────┘
                               │
              ┌────────────────┴────────────────┐
              │                                 │
         HTTP/1.1 & QUIC                   TCP & QUIC
         GateServer (8080)              ChatServer 4 节点集群
                                         (8090~8093 / 8190~8193)
              │                                 │
              │ gRPC                            │ gRPC
              ▼                                 ▼
         StatusServer (50052)  ◄───────── StatusServer (50052)
              │
              │ gRPC
              ▼
         VarifyServer (50051)
```

**各服务职责：**

| 服务 | 职责 | 协议 | 端口 |
|------|------|------|------|
| **GateServer** | HTTP 网关：登录/注册/媒体上传/朋友圈/路由转发 | HTTP/1.1, HTTP/3(QUIC) | 8080 |
| **StatusServer** | 在线状态管理、ChatServer 路由分配、gRPC 注册中心 | gRPC | 50052 |
| **ChatServer** | 聊天消息收发、好友关系、群组消息、多节点负载均衡 | TCP, QUIC, gRPC | 8090-8093, 8190-8193 |
| **VarifyServer** | 验证码服务（邮件发送） | gRPC | 50051 |

## 二、C++ 技术栈深度剖析

### 2.1 C++20 新特性在项目中的应用

MemoChat 全面使用 C++20，以下特性均在代码中有实际应用：

#### 2.1.1 概念（Concepts）与约束

CMakeLists.txt 中使用 `target_compile_features` 强制要求 C++20：

```cpp
target_compile_features(memochat_proto PUBLIC cxx_std_17)
set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
```

项目中通过 `requires` 约束模板参数类型，确保类型安全。

#### 2.1.2 `std::jthread` 与协程

ChatServer 中使用 `std::thread` 配合 `std::condition_variable` 实现优雅关闭：

```cpp
// server/ChatServer/ChatServer.cpp
std::condition_variable cond_quit;
std::mutex mutex_quit;
bool bstop = false;

// 信号处理线程 - 优雅关闭
boost::asio::io_context io_context;
boost::asio::signal_set signals(io_context, SIGINT, SIGTERM);
signals.async_wait([&server, &io_context](const boost::system::error_code& error, int signal_number) {
    if (!error) {
        server->Shutdown();
        io_context.stop();
    }
});
```

> **面试点**：虽然项目尚未使用 C++20 协程（co_await），但架构设计已经为协程化做了铺垫（所有 IO 操作均为异步回调）。

#### 2.1.3 `std::format` 与字符串处理

项目使用 `std::format`（C++20）进行高效字符串格式化，替代旧有的 `printf` / `ostringstream`：

```cpp
// 项目中大量使用
std::snprintf(buf, sizeof(buf), "%04d%02d%02d",
    local_tm.tm_year + 1900, local_tm.tm_mon + 1, local_tm.tm_mday);
```

#### 2.1.4 `std::span` 与视图

C++20 的 `std::span` 用于安全地传递数组视图，避免裸指针 + 长度的手动管理。

### 2.2 Boost.Asio 深度剖析

Boost.Asio 是整个服务端异步网络的核心。

#### 2.2.1 核心概念：Proactor 模式

Boost.Asio 采用 **Proactor 模式**（异步非阻塞 IO 模型）：

- **Reactor 模式**（select/poll/epoll）：应用主动询问内核 IO 是否就绪，eg. epoll_wait
- **Proactor 模式**（IOCP/epoll+异步回调）：内核完成 IO 后主动通知应用

```
  发起异步读 ──► 内核等待数据 ──► 数据就绪通知应用 ──► 调用回调
  (async_read)     (kernel)         (epoll/KQUEUE)      (handler)
```

Asio 的 `async_read`、`async_write` 等函数底层使用 epoll（Linux）/ IOCP（Windows）/ KQUEUE（macOS）。

#### 2.2.2 Asio 的 IO 上下文（io_context）

Asio 有两种运行模式：

**1. 多人共用一个 io_context（推荐生产）**：
```cpp
// ChatServer 使用一个 io_context 供所有 socket 共用
boost::asio::io_context io_context;
// 多个线程运行同一个 io_context（线程安全）
std::vector<std::thread> threads;
for (int i = 0; i < 4; ++i) {
    threads.emplace_back([&io_context]() { io_context.run(); });
}
```

**2. AsioIOServicePool：多 io_context 池化**：
ChatServer 使用 AsioIOServicePool 管理多个 io_context，减少锁竞争：
```cpp
// ChatServer/AsioIOServicePool.h - 每个 ChatServer 节点有多个 io_context
// 每个 io_context 由独立线程驱动，避免所有 session 争抢同一个线程调度器
class AsioIOServicePool {
    std::vector<std::unique_ptr<boost::asio::io_context>> _io_contexts;
    std::vector<std::thread> _ threads;
};
```

> **面试点**：为什么需要多个 io_context？
> - 单线程 Asio 的事件循环是串行的，高并发下回调函数之间相互阻塞
> - 多 io_context 池化可以分散负载，每个 io_context 由独立线程驱动
> - ChatServer 中每个 TCP 连接绑定到一个 io_context，QUIC 同理

#### 2.2.3 Boost.Beast 的 HTTP 处理

GateServer 使用 Boost.Beast 解析 HTTP 请求：

```cpp
// GateServer/HttpConnection.cpp
http::async_read(_socket, _buffer, _request,
    [self](beast::error_code ec, std::size_t bytes_transferred) {
        // 回调中处理解析后的请求
        self->HandleReq();
    });
```

Beast 是 Boost 的 HTTP 库，底层基于 Asio 的异步 IO。`http::async_read` 使用 Asio 的异步 IO 模型，从 socket 异步读取数据并解析为 HTTP 请求结构体。

#### 2.2.4 ASIO 对比 select/poll/epoll

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| 最大 fd 限制 | 1024（FD_SETSIZE） | 无限制 | 无限制 |
| 时间复杂度 | O(n) | O(n) | O(1) per event |
| 事件复杂度 | 每次都要遍历全部 fd | 每次都要遍历全部 fd | 只返回就绪的 fd |
| 适用场景 | fd 少 | fd 中等 | fd 庞大（数万） |
| 跨平台 | 全部 | Linux | Linux only |

Asio 底层根据平台自动选择最优后端：Linux 用 epoll，Windows 用 IOCP，macOS 用 KQUEUE。

### 2.3 Protobuf 与 gRPC

#### 2.3.1 Protocol Buffers 序列化原理

Protobuf 是二进制序列化协议，相比 JSON 有显著优势：

| 指标 | JSON | Protobuf |
|------|------|---------|
| 体积 | 大（文本，有键名） | 小（二进制，无键名，用 tag） |
| 序列化速度 | 中等 | 快（直接内存操作，无需字符串解析） |
| 反序列化速度 | 慢 | 快 |
| 可读性 | 高 | 低（需要 proto 定义） |
| 跨语言 | 广泛 | 需要生成代码 |
| Schema 演进 | 灵活 | 需要严格管理 |

**Protobuf Tag 机制**（面试高频）：
```proto
message User {
    int32 uid = 1;      // tag=1, wire_type=0 (varint)
    string name = 2;    // tag=2, wire_type=2 (length-delimited)
    bytes avatar = 3;   // tag=3, wire_type=2 (length-delimited)
}
```

每个字段编码为：`tag << 3 | wire_type` + length + data
- `wire_type=0`：varint（用于 int32, int64, uint32 等）
- `wire_type=2`：长度分隔（用于 string, bytes, 嵌套消息）
- `wire_type=5`：固定32位（用于 float, fixed32）

#### 2.3.2 gRPC 工作原理

gRPC 基于 HTTP/2 协议，使用 Protobuf 作为序列化格式：

```
客户端 ──HTTP/2──► gRPC Server Stub ──► 业务处理
      ◄──流式响应─┘
```

**四种 RPC 模式：**

1. **一元调用（Unary）**：客户端发一个请求，服务器返回一个响应（像普通函数调用）
2. **服务器流式（Server Streaming）**：客户端发一个请求，服务器返回多个响应
3. **客户端流式（Client Streaming）**：客户端发多个请求，服务器返回一个响应
4. **双向流式（Bidirectional Streaming）**：客户端和服务器都可以连续发送多条消息

MemoChat 的 StatusServer 使用 gRPC 提供 `GetChatServer` 和 `Login` 服务。

```cpp
// StatusServer/StatusServiceImpl.cpp
Status GetChatServer(ServerContext* context,
    const GetChatServerReq* request,
    GetChatServerRsp* reply) override {
    // 查询 Redis 获取最优 ChatServer 节点
    // 返回 host:port + token
}
```

> **面试点**：gRPC vs RESTful
> - gRPC 使用 HTTP/2，支持多路复用（一个 TCP 连接上并行多个请求）
> - Protobuf 二进制序列化，效率远高于 JSON
> - RESTful 使用 HTTP/1.1 或 HTTP/2，但 body 通常是 JSON
> - gRPC 支持双向流，RESTful 不支持
> - gRPC 需要 .proto 文件定义接口，RESTful 通常 OpenAPI/Swagger

## 三、数据库技术栈深度剖析

### 3.1 PostgreSQL 深度剖析

#### 3.1.1 为什么 IM 系统选择 PostgreSQL

PostgreSQL 是功能最强大的开源关系型数据库，非常适合 IM 的核心业务数据：

- **强 ACID**：好友关系、群组成员关系必须保证一致性
- **丰富索引**：B-tree、Hash、GIN、GiST、BRIN 等多种索引类型
- **JSON 支持**：PostgreSQL 9.3+ 支持 JSONB，可以存储半结构化数据
- **JSON Path 查询**：可以像 MongoDB 一样查询 JSON 字段
- **CTE（公用表表达式）**：递归查询组织架构
- **窗口函数**：在线统计、排名

MemoChat 在 PostgreSQL 中存储：用户表、好友关系表、群组表、群组成员表、会话表。

#### 3.1.2 PostgreSQL 索引类型与选型

```sql
-- B-tree 索引（默认，最常用）：适用于等值查询和范围查询
CREATE INDEX idx_user_email ON "user"(email);
CREATE INDEX idx_friend_uid1 ON friend(uid1);

-- GIN 索引：适用于全文搜索、JSONB 数组包含查询
CREATE INDEX idx_user_name_gin ON "user" USING gin(name gin_trgm_ops);

-- BRIN 索引：适用于按自然顺序范围查询（如时间序列）
CREATE INDEX idx_msg_time ON chat_message USING BRIN(created_at);
```

> **面试点**：B-tree vs Hash 索引
> - B-tree：平衡多路搜索树，支持 >, <, =, BETWEEN, LIKE 前缀匹配，范围查询
> - Hash：只支持 = 精确匹配，无法做范围查询
> - PostgreSQL 的 Hash 索引不支持多列
> - 好友关系 uid1 的精确查找用 Hash 更快，但实际项目中多用 B-tree（因为 B-tree 更通用，且支持范围扫描）

#### 3.1.3 libpqxx 连接池

ChatServer 使用 libpqxx 连接 PostgreSQL：

```cpp
// ChatServer/PostgresPool.h
class PostgresPool {
    std::queue<pqxx::connection*> _pool;
    std::mutex _mutex;
    std::condition_variable _cond;
};
```

> **面试点**：为什么需要数据库连接池？
> - 每次 SQL 操作都建立 TCP 连接开销巨大（3次握手 + SSL 握手 + 认证）
> - 数据库连接数有限，不能每个请求创建一个连接
> - 连接池复用已建立的连接，避免频繁建立/销毁
> - 典型的连接池大小：CPU 核数的 2~4 倍

#### 3.1.4 连接字符串中的 search_path

```cpp
// PostgresMgr.cpp
" options=-csearch_path=" + schema + ",public"
```

设置默认 schema 为 `memo`，所有表都在 `memo` schema 下，避免跨库冲突。

### 3.2 Redis 深度剖析

#### 3.2.1 Redis 在 IM 系统中的核心作用

MemoChat 使用 Redis 做五件事：

1. **会话缓存**：`USER_SESSION:{uid}` 存储登录态 token 和 ChatServer 节点信息
2. **在线状态**：`USER_IP:{uid}` 记录用户在哪台 ChatServer
3. **发布订阅**：好友上线通知、消息投递通知
4. **分布式锁**：`LOCK:{resource}` 保护关键操作的并发安全
5. **计数器和限流**：`ONLINE_COUNT` 在线人数统计

#### 3.2.2 Redis 数据结构选择

```
String    ─── 用户 Session (JSON 序列化的用户信息)
Hash      ─── 在线用户集合 (HMGET/HSET 批量操作)
Set       ─── ChatServer 节点的在线用户集合
Sorted Set ── 延迟队列、优先级消息
Pub/Sub   ─── 实时通知（好友上线、消息推送）
Stream    ─── 消息队列替代方案（支持持久化、消费确认）
Bitmap    ─── 日活统计、在线状态位图
```

> **面试点**：String vs Hash vs Set 选型
> - `String`：单个值，简单 KV，序列化后存复杂对象
> - `Hash`：`HGETALL` 一次取所有字段，适合对象存储（避免多次 RTT）
> - `Set`：无序不重复集合，适合标签、好友列表（`SISMEMBER` O(1) 判断）
> - **MemoChat 选型**：Session 用 String + JSON，缓存命中判断用 Hash，在线用户用 Set

#### 3.2.3 Redis 发布订阅（Pub/Sub）

```cpp
// 订阅消息通知频道
redisCommand(conn, "SUBSCRIBE user_notify_%d", uid);

// 发布好友上线通知
redisCommand(conn, "PUBLISH user_notify_%d {\"event\":\"friend_online\"}", friend_uid);
```

> **面试点**：Redis Pub/Sub 的局限性
> - **不持久化**：消息在内存中，订阅者不在线则消息丢失
> - **不保证送达**：客户端断线重连后，丢失的消息不会补发
> - **不分区**：单节点 Pub/Sub，不支持 Redis Cluster
> - **替代方案**：Redis Stream（支持持久化、消费者组 ACK）或 Kafka

#### 3.2.4 Redis 分布式锁（Redlock 简化版）

```cpp
// ChatServer/DistLock.cpp
// 使用 SET NX + EX 实现分布式锁
redisCommand(conn, "SET %s %s NX EX %d", lock_key, value, ttl);
```

> **面试点**：Redis 分布式锁的坑
> - **锁过期**：任务执行时间 > 锁 TTL，锁自动释放，其他进程获取同一把锁（解决方案：锁续期）
> - **非原子性**：`GET` + `DEL` 不是原子操作（解决方案：Lua 脚本或 SET NX + 值校验）
> - **主从切换**：主节点加锁成功但未同步到从节点时主节点宕机（解决方案：Redlock 算法，多节点多数投票）
> - MemoChat 中用 Redis 分布式锁保护 Snowflake ID 生成

### 3.3 MongoDB 深度剖析

#### 3.3.1 为什么聊天消息用 MongoDB

聊天消息的特点：
- **写入量大**：每个用户每天产生数百到数千条消息
- **schema 灵活**：文本消息、图片消息、视频消息结构不同
- **查询模式简单**：通常按时间倒序查，不需要复杂 JOIN
- **半结构化**：每条消息可以有附加字段（引用、回复等）

PostgreSQL 的写入放大和高并发锁开销在海量消息场景下是瓶颈。

#### 3.3.2 MongoDB 的 Journal（Write Ahead Log）

MongoDB 默认启用 Journal（预写日志），每次写操作：
1. 先写 Journal（持久化到磁盘）
2. 再修改内存中的数据
3. 后台定期将脏页刷入数据文件

```bash
# Journal 配置
storage.journal.enabled = true  # 默认开启
```

> **面试点**：MongoDB vs PostgreSQL 写入路径对比
> - PostgreSQL：WAL → 数据页 → Buffer Pool（Double Write Buffer 防 partial write）
> - MongoDB：Journal → 数据文件（MMAPv1 或 WiredTiger 引擎）
> - WiredTiger 默认使用 B-tree 存储，有自己的 MVCC 和 checkpoint 机制

#### 3.3.3 mongocxx 驱动使用

```cpp
// ChatServer/MongoDao.cpp
mongocxx::client client{mongocxx::uri{mongo_uri}};
mongocxx::database db = client[database_name];
mongocxx::collection coll = db[collection_name];

// 插入消息
bsoncxx::document::value doc = bsoncxx::builder::stream::document{}
    << "from_uid" << from_uid
    << "to_uid" << to_uid
    << "content" << content
    << "created_at" << bsoncxx::types::b_date{std::chrono::system_clock::now()}
    << bsoncxx::builder::stream::finalize;
coll.insert_one(doc.view());
```

### 3.4 数据库选型对比（面试高频）

| 维度 | PostgreSQL | MongoDB | Redis |
|------|-----------|---------|-------|
| 数据模型 | 关系型 | 文档型 | KV / 多种结构 |
| 一致性 | 强 ACID | 默认最终一致（单文档 ACID） | 弱一致（可配置） |
| 扩展性 | 垂直扩展为主 | 水平分片（Sharding） | 内存受限，Cluster 模式 |
| 写入性能 | 中等（事务开销） | 高（B-tree，无事务开销） | 极高（纯内存） |
| 复杂查询 | 强（JOIN、聚合、子查询） | 有限（aggregation pipeline） | 有限（Lua 脚本） |
| 适用场景 | 核心业务数据 | 海量消息、灵活 schema | 缓存、会话、实时状态 |

> **面试点**：MemoChat 的多数据库分工策略
> - PostgreSQL 存"元数据"（谁是谁的好友、属于哪个群组）
> - MongoDB 存"消息内容"（聊天记录）
> - Redis 存"实时状态"（谁在线、当前在哪个 ChatServer）
> - 这就是经典的 **Polyglot Persistence（多语言持久化）** 架构

## 四、消息队列技术栈深度剖析

### 4.1 Kafka（Redpanda）深度剖析

#### 4.1.1 为什么 IM 需要消息队列

ChatServer 是有状态的（每个节点管理自己的 TCP 连接），但服务间需要协作：

```
用户 A (在 ChatServer-1) ──发送消息──► ChatServer-1
                                             │
                                        Kafka Topic
                                             │
                                    ┌────────┴────────┐
                                    │ 消费此消息       │
                              ChatServer-2      ChatServer-3
                            (目标用户 B)      (目标用户 C)
```

如果用户 B 不在 ChatServer-1 上，ChatServer-1 需要通过 Kafka 将消息投递出去，让 B 所在的节点来处理通知。

#### 4.1.2 Kafka 核心概念

| 概念 | 说明 |
|------|------|
| **Topic** | 消息的逻辑分类，eg. `chat.private.v1` 私聊消息 |
| **Partition** | Topic 的物理分区，支持并行消费和水平扩展 |
| **Producer** | 消息生产者，ChatServer 往 Kafka 写消息 |
| **Consumer** | 消息消费者，ChatServer 消费其他节点转发的消息 |
| **Consumer Group** | 消费者组，同一组内不同 Consumer 消费不同 Partition |
| **Offset** | 消费者在 Partition 中的消费位置 |
| **Leader/Replica** | 每个 Partition 有多个副本，Leader 处理读写 |

#### 4.1.3 Kafka 写入流程（面试高频）

```
Producer ──1. 序列化消息──► 计算 Partition ──2. 发送到 Leader Broker──► 
                                                      │
                                           ┌──────────┴──────────┐
                                           │ Leader 写入内存      │
                                           │ (OS Page Cache)      │
                                           └──────────┬──────────┘
                                           等待 ISR 同步 ACK
                                           (In-Sync Replicas)
                                                      │
                                           ┌──────────┴──────────┐
                                           │ Follower 拉取并同步 │
                                           │ 所有 ISR 确认后      │
                                           │ Leader 返回 ACK     │
                                           └──────────┬──────────┘
                                                      │
                                           3. Producer 收到 ACK
```

> **面试点**：acks 参数的影响
> - `acks=0`：Producer 不等待确认，最快但最不安全，可能丢消息
> - `acks=1`：Leader 确认写入即返回，不等 Follower 同步（默认）
> - `acks=all`（或 `-1`）：所有 ISR 都确认后才返回，最安全但最慢
> - MemoChat 的 ChatServer 配置 `acks=all` 并开启 `enable.idempotence=true` 保证exactly-once

#### 4.1.4 Kafka vs RabbitMQ 对比

| 维度 | Kafka | RabbitMQ |
|------|-------|---------|
| 架构模型 | 日志追加（Partition） | 队列（Queue） |
| 消息持久化 | 默认持久化到磁盘 | 可配置，需要手动开启 |
| 消息回溯 | 支持（通过 offset 回溯） | 不支持 |
| 消费模式 | 发布订阅（Consumer Group） | 点对点 + 发布订阅 |
| 顺序保证 | Partition 内有序 | 队列内有序（单队列） |
| 吞吐量 | 百万级/秒 | 万级/秒 |
| 延迟 | 毫秒级 | 微妙~毫秒级 |
| 适用场景 | 大数据流、日志、事件溯源 | 任务队列、异步任务 |

> **面试点**：MemoChat 的双 MQ 策略
> - **Kafka**：事件总线，消息分发（ChatServer 间转发聊天消息）
> - **RabbitMQ**：任务队列，异步操作（发送邮件、推送通知等异步 side-effect）
> - Kafka 适合高吞吐的流式消息，RabbitMQ 适合需要可靠投递的任务

### 4.2 RabbitMQ 深度剖析

#### 4.2.1 RabbitMQ 核心概念

| 概念 | 说明 |
|------|------|
| **Exchange** | 消息交换机，根据 routing_key 路由到 Queue |
| **Queue** | 消息队列，FIFO |
| **Binding** | Exchange 和 Queue 的绑定关系 |
| **Routing Key** | 路由键，Exchange 根据它决定消息去哪个 Queue |
| **Message acknowledgment** | 消费确认，消息处理完后 ACK |
| **Dead Letter Exchange (DLX)** | 死信队列，消息消费失败后进入 DLX |

#### 4.2.2 Exchange 类型

MemoChat 使用两种 Exchange：

```
Direct Exchange (memochat.direct)
  routing_key = "chat.notify" ──► Queue: chat_notify_queue

Dead Letter Exchange (memochat.dlx)
  消费失败的 message ──► DLX ──► DLQ (Dead Letter Queue)
```

- **Direct Exchange**：精确匹配 routing_key
- **Fanout Exchange**：广播到所有绑定的 Queue
- **Topic Exchange**：通配符匹配（`*` 匹配一个词，`#` 匹配零个或多个词）

#### 4.2.3 RabbitMQ 消息可靠性保障

```cpp
// ChatServer/RabbitMqTaskBus.cpp
// 消费确认模式
channel.basic_qos(prefetch_count);  // 每次只给消费者 prefetch_count 条消息
auto delivery_tag = envelope.get_delivery_tag();
channel.basic_ack(delivery_tag);    // 处理完后 ACK
// 如果失败：channel.basic_nack(delivery_tag, false, true); // requeue
```

> **面试点**：如何保证消息不丢失？
> - **生产者端**：`publisher confirms` 机制，生产者确认消息被 Broker 接收
> - **Broker 端**：开启持久化（durable queue + persistent message）
> - **消费者端**：`auto_ack=false`，手动 ACK，消息处理成功后再 ACK
> - **死信队列**：消息多次消费失败后进入 DLQ，避免无限重试

## 五、QUIC 协议深度剖析

### 5.1 为什么 IM 需要 QUIC

**TCP 的三次握手 + TLS 的握手延迟**：

```
TCP 三次握手：           1 RTT
TLS 1.2 握手：           2 RTT（Full Handshake）
HTTP 请求/响应：         1 RTT
───────────────────────────────
总计：                    4 RTT 才能发出第一个字节！
```

对于移动网络和高延迟网络，4 RTT 是不可接受的。

### 5.2 QUIC 协议核心原理

QUIC 是基于 UDP 的传输协议，解决了 TCP 的多个痛点：

#### 5.2.1 0-RTT / 1-RTT 建立连接

```
TCP + TLS 1.3:
  ClientHello ──────────────────────────────►
                ◄─────────────────────────── ServerHello + Certificate + ...
  Finished ─────────────────────────────────►
  Request ──────────────────────────────────►
  Response ◄─────────────────────────────────
  总计: 2 RTT (TLS 1.3 Full Handshake) + 1 RTT (TCP handshake) = 3 RTT
  
  如果之前连接过 (TLS Session Resumption):
  Finished ──0-RTT──► Request ──────────────────────────────────►
  Response ◄─────────────────────────────────
  总计: 1 RTT (0-RTT 不验证服务器身份，有重放攻击风险)

QUIC:
  ClientHello ────► (含加密参数)
                    ◄─── ServerHello + 加密数据 + ServerFinished
  Encrypted Request ──►
  Encrypted Response ◄─
  总计: 1 RTT
  
  如果之前连接过 (QUIC 0-RTT):
  Early Data ─────────► (直接发送加密数据，无需等待 ServerHello)
  总计: 0 RTT
```

> **面试点**：QUIC 0-RTT 的安全权衡
> - 0-RTT 使用之前会话的密钥加密，无法提供前向保密（Forward Secrecy）
> - 存在重放攻击风险（恶意重发相同 0-RTT 数据包）
> - 适用于对延迟敏感且可以容忍一定风险的应用（IM、新闻推送）
> - 重要操作（支付）不适合 0-RTT

#### 5.2.2 多路复用（避免 TCP 队头阻塞）

**TCP 队头阻塞（Head-of-Line Blocking）**：
```
TCP 是字节流协议，一个 TCP 连接上：
  GET /api/msg ──► 丢包 ──► TCP 重传
                         ──► GET /api/user ──► 必须等待重传完成
  
  HTTP/1.1: 多个 TCP 连接（浪费资源）
  HTTP/2:   一个 TCP 连接上多路复用（但还是受 TCP HOL 阻塞）
```

**QUIC 的解决**：
```
每个 Stream 是独立的，丢包只影响丢包所在的 Stream
  Stream 1: GET /api/msg ──► 丢包（只阻塞 Stream 1）
  Stream 2: GET /api/user ──► 正常收到（不等待 Stream 1）
  
  Stream 3: GET /api/friend ──► 正常收到
```

#### 5.2.3 连接迁移（Connection Migration）

TCP 连接由四元组（src_ip, src_port, dst_ip, dst_port）标识。移动设备切换网络（WiFi → 4G）时 IP 变化，TCP 连接必须重建。

QUIC 使用 **Connection ID** 标识连接：
```
Connection ID: abc123 (独立于 IP)
  WiFi:  192.168.1.x ──► QUIC (CID=abc123) ──► Server
  切换到 4G
  4G:    10.0.0.x    ──► QUIC (CID=abc123) ──► Server
  (同一连接，无需重建!)
```

#### 5.2.4 msquic 在项目中的应用

ChatServer 的 `QuicSession` 基于 msquic 实现：

```cpp
// ChatServer/QuicSession.h
class QuicSession : public CSession {
    // 继承 CSession 的所有业务逻辑
    // SendCallback: msquic 回调，用于发送数据
    // CloseCallback: 连接关闭回调
};

// ChatServer/QuicChatServer.h
// 启动 QUIC 监听
boost::asio::io_context quic_io_context;
// 每个 ChatServer 节点支持一个 TCP 端口 + 一个 QUIC 端口
// chatserver1: TCP=8090, QUIC=8190
```

配置示例（config.ini）：
```ini
[Quic]
Alpn=memochat-chat
PfxFile=server-schannel3.pfx  # Windows SChannel 证书
CertFile=server-cert.pem
PrivateKeyFile=server-key.pem
```

> **面试点**：msquic vs ngtcp2
> - **msquic**：微软开源 QUIC 库，基于 Windows SChannel，Windows 友好
> - **ngtcp2 + nghttp3**：社区开源，Linux/macOS 友好，支持 BoringSSL/OpenSSL
> - MemoChat 两个都支持编译，通过宏切换：`MEMOCHAT_ENABLE_HTTP3`

## 六、服务间通信与微服务架构

### 6.1 四种通信模式在项目中的实际应用

MemoChat 存在四种服务间通信方式：

| 模式 | 例子 | 特点 |
|------|------|------|
| **同步 HTTP/REST** | 客户端 → GateServer | 简单，请求-响应 |
| **gRPC** | StatusServer ↔ ChatServer | 高效，二进制，双向流 |
| **TCP 长连接** | 客户端 ↔ ChatServer | 实时，低延迟 |
| **消息队列** | GateServer → ChatServer (Kafka) | 异步，解耦 |

### 6.2 为什么 ChatServer 需要 4 个节点

```
ChatServer-1  (8090/8190) ── 持有部分用户的长连接
ChatServer-2  (8091/8191) ── 持有部分用户的长连接
ChatServer-3  (8092/8192) ── 持有部分用户的长连接
ChatServer-4  (8093/8193) ── 持有部分用户的长连接
```

- **容量扩展**：每个节点能承载的 TCP 连接数有上限（Linux 默认 1024，可调）
- **地理位置**：不同节点服务不同地域的用户
- **故障隔离**：一个节点崩溃不影响其他节点的用户

### 6.3 如何将用户路由到正确的 ChatServer 节点

```
客户端登录流程：
1. HTTP POST /login → GateServer
2. GateServer → StatusServer (gRPC): GetChatServer(uid)
3. StatusServer 查询 Redis: "user_ip:{uid}" → ChatServer 节点信息
4. StatusServer 返回 ChatServer host:port + 临时 token
5. 客户端 → ChatServer (TCP/QUIC): 用 token 建立长连接
6. ChatServer → Redis: SET "user_ip:{uid}" = "chatserver1"
```

### 6.4 微服务间的服务发现

**静态配置模式**（当前 MemoChat 使用）：
```ini
[Cluster]
DiscoveryMode=static
Nodes=chatserver1,chatserver2,chatserver3,chatserver4
```

每个 ChatServer 配置自身和对端节点的地址。但这种方式的问题是：
- 扩缩容需要手动改配置
- 节点故障需要人工介入

**更优方案：动态服务发现**：
```
Consul / etcd / ZooKeeper
  ChatServer 启动时注册自己（IP + Port + 健康状态）
  StatusServer 从注册中心获取可用 ChatServer 节点列表
  故障检测：注册中心心跳探测，不健康节点自动注销
```

### 8.5 Qt/QML 信号槽机制原理

```cpp
// 核心原理：元对象系统（Meta-Object System）
// Qt 的 QObject + QMetaObject 实现运行时反射和信号槽

// moc (Meta-Object Compiler) 预处理阶段为每个 QObject 生成
// moc_*.cpp 文件，包含：
//   - QMetaObject 元数据（信号/槽/属性列表）
//   - qt_metacall(): 根据 index 调用对应的槽函数
//   - qt_static_metacall(): 模板实现，编译期优化

// 信号触发时：
//   1. QMetaObject::activate() 遍历所有连接（queued/auto/direct）
//   2. 通过 QMetaCallEvent 事件队列（QueuedConnection）或直接调用
```

> **面试点**：Qt 信号槽 vs 回调函数
> - **信号槽**：松耦合，发送者不知道接收者是谁，多对多连接
> - **回调函数**：紧耦合，一对一，调用者必须知道函数指针
> - Qt 信号槽支持跨线程（QueuedConnection，自动事件队列）
> - 性能上 direct 模式几乎零开销，queued 模式有事件队列入队开销

## 九、架构设计模式与最佳实践

### 9.1 设计模式在项目中的应用

| 模式 | 应用位置 | 作用 |
|------|---------|------|
| **单例模式** | `Singleton<HttpMgr>` `Singleton<RedisMgr>` | 全局唯一资源管理器 |

| **管道/过滤器** | `HttpConnection::HandleReq()` | 请求处理管道 |

### 9.2 CQRS 模式（命令查询职责分离）

```
写路径（Command）：
  客户端发送消息 → GateServer → ChatServer → PostgreSQL (好友关系)
                                              → MongoDB (消息内容)
                                              → Redis (在线状态)

读路径（Query）：
  客户端拉取历史 → GateServer → MongoDB (聊天记录)
                             → Redis (会话列表缓存)
```

### 9.3 Saga 模式（分布式事务）

用户注册流程涉及多个服务：
```
1. GateServer 写入 PostgreSQL (用户表)
2. GateServer 写入 Redis (Session)
3. VarifyServer 发送验证码邮件
```

如果第 3 步失败，需要回滚前两步——这就是 Saga 编排。MemoChat 通过 Try-Catch + 补偿事务实现简化版 Saga。

## 十、面试高频问题与回答要点

### Q1：你们的系统能承载多少用户？

**回答思路**：
- 理论计算：4 个 ChatServer × 每节点 10000 连接 = 40000 并发长连接
- 每个 ChatServer 使用多个 io_context 线程，每个处理约 2000-3000 连接
- HTTP 网关 GateServer 无状态，可水平扩展
- Redis 内存足够可支撑百万级 Key

**实际压测数据**（参见 `local-loadtest-cpp/README.md`）：
- HTTP 登录：单节点 5000+ QPS
- TCP 长连接登录：单节点 3000+ QPS
- QUIC 长连接：与 TCP 相近，优势在于高延迟网络

### Q2：如何保证消息不丢不重？

**回答要点**：

```
发送方保障（不丢）：
  1. ChatServer 写入 MongoDB 后再 ACK 客户端
  2. Kafka 消息持久化 (acks=all + durable topic)
  3. 消费者手动 ACK，失败后 requeue 或 DLQ

发送方保障（不重）：
  1. Kafka 开启 idempotence（每个消息带唯一 PID+seq）
  2. 客户端消息带唯一 msg_id，Server 做幂等去重
  3. MongoDB 消息 ID 唯一索引防止重复写入

接收方保障（可靠投递）：
  1. TCP/QUIC 可靠传输（ARQ 自动重传）
  2. 消息到达后先 ACK，异步写入存储
  3. 离线消息通过拉取补齐
```

### Q3：如何实现消息顺序性？

**回答要点**：
- 单聊消息：按 msg_id 排序（全局递增 ID），严格有序
- 群聊消息：在每个群内有序（通过 GlobalSeq 或群内序列号）
- 跨群/跨人会话：按时间戳倒序展示，允许轻微乱序
- 技术实现：Snowflake ID（时间戳 + datacenter + worker + 序列号）

```cpp
// Snowflake ID 结构：
// | timestamp(41bit) | datacenter(5bit) | worker(5bit) | seq(12bit) |
// 最多 32 个数据中心 × 32 台机器 × 每毫秒 4096 个 ID
// 支持约 69 年不冲突
```

### Q4：Redis 缓存与数据库一致性如何保证？

**回答要点**（Cache-Aside 模式）：

```
读：
  1. 先查 Redis（热点数据）
  2. Redis 未命中 → 查 PostgreSQL
  3. 将结果写回 Redis（设置 TTL）

写：
  1. 先更新 PostgreSQL（主数据库）
  2. 删除 Redis Key（而非更新，因为并发下可能覆盖）
  3. 下次读请求会重新加载最新数据

为什么不更新缓存而是删除？
  → 写操作后删除，下次由读请求重建
  → 避免 ABA 问题（删除后又被设置为旧值）
  → 减少不一致窗口
```

### Q5：分布式锁的实现细节？

```cpp
// 错误的实现（先 GET 再 SET，不是原子操作）：
bool acquire = redisCommand(conn, "GET %s", lock_key) == nullptr;
if (acquire) {
    redisCommand(conn, "SET %s %s EX %d", lock_key, value, ttl); // 其他进程可能已设置！
}

// 正确的实现（原子 SET NX + EX）：
// SET lock_key unique_value NX EX 30
// NX: 不存在才设置（原子）
// EX 30: 30 秒过期
// 返回 OK 才算获取锁成功

// 释放锁（Lua 脚本，保证原子性）：
// if redis.call("GET", KEYS[1]) == ARGV[1] then
//     return redis.call("DEL", KEYS[1])
// else
//     return 0
// end
```

### Q6：为什么选择 gRPC 而不是 RESTful 做服务间通信？

```
1. 性能：Protobuf 二进制序列化，比 JSON 小 3-10 倍，解析快 10-100 倍
2. 效率：HTTP/2 多路复用，单 TCP 连接并行多个 RPC
3. 类型安全：.proto 文件定义接口，编译期检查
4. 双向流：ChatServer 可以主动推送消息给 StatusServer
5. 代码生成：自动生成客户端/服务器骨架，减少样板代码
6. 自描述：proto 文件即文档，跨语言团队协作更顺畅
```

### Q7：微服务拆分的考量是什么？

```
拆分依据（按业务边界）：
  GateServer  ─── 接入层（HTTP 网关，认证入口）
  StatusServer ── 状态管理（无状态，可水平扩展）
  ChatServer   ── 核心业务（有状态，连接密集）
  VarifyServer ── 验证码（IO 密集，可独立扩缩）

不拆分过细的原因：
  → 保持服务间延迟可控（本地函数调用 vs 网络调用）
  → 减少运维复杂度（4 个服务 vs 40 个服务）
  → 平衡开发效率和系统复杂度
```

### Q8：如何做服务降级和熔断？

```
服务降级：
  1. ChatServer 不可用 → GateServer 返回 "服务器繁忙，稍后重试"
  2. 验证码服务不可用 → 降级为 Mock 模式（开发环境）
  3. MongoDB 不可用 → 消息写入内存队列，暂存后重试

熔断策略：
  1. 连续 N 次超时 → 开启熔断（快速失败，不发请求）
  2. 熔断期间每 M 秒放行一个探测请求
  3. 探测成功 → 关闭熔断，恢复正常
  4. 探测失败 → 延长熔断时间

MemoChat 中的体现：
  - 登录失败后本地缓存 token（降级）
  - ChatServer 故障时 StatusServer 返回备用节点（降级）
  - Kafka/RabbitMQ 连接失败时的重连退避策略（熔断）
```

### Q9：C++ 中 RAII 模式如何防止内存泄漏？

```cpp
// GateServer 使用 boost::asio 的智能指针管理生命周期
// 所有异步回调都通过 shared_from_this() 保持对象存活
class HttpConnection : public std::enable_shared_from_this<HttpConnection> {
    void Start() {
        auto self = shared_from_this();  // 捕获自身 shared_ptr
        http::async_read(_socket, _buffer, _request,
            [self](beast::error_code ec, std::size_t) {
                self->HandleReq();  // self 保证 HttpConnection 在回调期间不被析构
            });
    }
};
```

### Q10：Windows 平台的特殊考量？

```
1. IOCP vs epoll：
   - Windows 用 IOCP（I/O Completion Ports），Linux 用 epoll
   - Asio 统一抽象，自动选择最优后端
   - Boost.Asio 保证跨平台代码一致

2. 路径分隔符：
   - Windows: \（反斜杠）
   - Linux: /（正斜杠）
   - 代码中使用 std::filesystem::path 或 QDir 自动处理

3. MSVC vs GCC/Clang：
   - CMakeLists.txt 中用 IF(MSVC) 分支处理编译选项差异
   - /external:I 外部包含目录（MSVC 特有的合规模式警告控制）

4. 网络字节序：
   - Windows/Linux 都是大端/big-endian 网络字节序
   - 跨平台代码统一用 ntohl/htonl 处理字节序

5. SSL/TLS：
   - Windows 用 SChannel（系统自带）
   - Linux 用 OpenSSL
   - msquic 自动适配，QUIC 证书配置差异化处理
```

## 十一、系统瓶颈分析与优化思路

### 11.1 当前架构的瓶颈

| 瓶颈点 | 原因 | 优化方向 |
|--------|------|---------|
| ChatServer 单节点连接数 | Linux fd 限制 / 内存 | 增加节点数、使用连接池 |
| PostgreSQL 写入 | 事务 + 索引维护开销 | 分库分表、异步刷盘 |
| Redis 内存 | 全量在线用户数据 | 热冷数据分离、冷数据落盘 |
| Kafka 吞吐 | 单 Partition 顺序写 | 多 Partition + 消费者并行 |
| GateServer HTTP | 单点入口 | Nginx/LB 多 GateServer 负载均衡 |

### 11.2 水平扩展策略

```
GateServer: 无状态，直接增加节点，前面加 Nginx 负载均衡
StatusServer: 无状态，增加节点，客户端随机选择
ChatServer: 有状态，增加节点，修改路由策略（一致性哈希）
  一致性哈希：
    原来: uid % 4 = ChatServer 节点
    优化后: 一致性哈希环，扩缩容时只影响少量用户
            （只迁移 ~1/n 的数据，而不是全部）
```

### 11.3 数据库读写分离

```
PostgreSQL:
  主库 (Master) ──► 同步复制 ──► 从库 (Read Replica) × N
  写操作 ──► Master
  读操作 ──► Replica（降低 Master 压力）
  
  适用场景：好友列表、群组信息等读多写少的数据
  不适用：在线状态（Redis 更适合）、聊天消息（MongoDB）
```

## 十二、技术选型面试回答模板

### 12.1 "为什么选择 C++ 作为后端语言？"

```
项目最初的需求是高并发低延迟的即时通讯服务器。
相比 Go/Java/C#，C++ 的优势在于：

1. 内存控制：手动内存管理 + RAII，没有 GC 停顿
2. 性能天花板：C++ 程序的 CPU 和内存效率是最高的，
   对于 IO 密集型的 IM 服务器尤其重要
3. 成熟的异步生态：Boost.Asio 提供了久经验证的异步 IO
4. 硬件亲和性：直接操作 socket，无需 JVM/.NET 抽象层

当然 C++ 的缺点是开发效率相对低，所以我们：
  - 使用 Drogon 框架减少样板代码
  - 依赖 vcpkg 管理依赖
  - 全面使用 C++20 现代语法保持代码可维护性

实际上在 2020 年后，C++ 的异步框架（Asio/Drogon）已经非常成熟，
很多知名 IM 系统（如 Discord 的语音后端）也使用 C++。
```

### 12.2 "为什么需要多种数据库？"

```
每种数据库都有其设计哲学和最适场景。

PostgreSQL 用于核心关系数据，因为：
  - 好友关系、群组成员必须保证强一致性
  - 复杂的 JOIN 查询（查某用户的所有共同好友）
  - 事务支持（好友申请需要原子地创建两条记录）

MongoDB 用于聊天消息，因为：
  - 消息量巨大（日活 1 万用户 × 100 条/天 = 100 万条/天）
  - schema 灵活（文本/图片/视频消息结构不同）
  - 不需要复杂 JOIN，按时间倒序查询为主

Redis 用于实时状态，因为：
  - 毫秒级延迟（查用户在线状态必须快）
  - 内存型存储，不需要持久化
  - 发布订阅机制天然适合好友上线通知

这就是 Polyglot Persistence——针对不同数据特性选择最适合的存储。
```

### 12.3 "你们的项目有什么亮点？"

```
1. 多协议接入：同一套后端服务，同时支持 HTTP/1.1、HTTP/3(QUIC)、TCP 长连接
   → 体现了对不同场景的适配能力（HTTP 用于管理类操作，TCP/QUIC 用于聊天）

2. 混合 MQ 架构：Kafka + RabbitMQ 分工明确
   → Kafka 处理高吞吐消息分发，RabbitMQ 处理可靠任务投递
   → 体现了对不同 MQ 特性的深刻理解

3. 完整的可观测性：spdlog + OTel + Prometheus + Grafana
   → 从日志、链路、指标三个维度监控服务
   → trace_id 全链路透传，可在 Grafana 中一键查看完整调用链

4. 4 节点 ChatServer 集群 + 静态路由
   → 展示了我对分布式系统的理解

5. C++20 + CMake + vcpkg 现代 C++ 开发栈
   → 代码现代、构建规范、跨平台支持
```
