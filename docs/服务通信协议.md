# MemoChat 服务通信协议

> 文档版本：2026-03-23

---

## 一、协议概述

MemoChat 采用多层协议体系，支持 HTTP/1.1、HTTP/3 (QUIC)、TCP 和 gRPC 四种通信协议。不同业务场景使用不同协议，以达到最优的性能和可靠性。

### 1.1 协议选择策略


| 业务场景    | 推荐协议     | 备选协议 | 说明         |
| ------- | -------- | ---- | ---------- |
| 用户登录/注册 | HTTP/1.1 | -    | 简单可靠，防火墙友好 |
| 验证码获取   | HTTP/1.1 | -    | 一次性请求      |
| 聊天消息    | TCP      | QUIC | 低延迟，可靠传输   |
| 文件上传/下载 | HTTP/1.1 | -    | 支持断点续传     |
| 音视频通话   | QUIC     | TCP  | 低延迟，抗丢包    |
| 服务间通信   | gRPC     | -    | 类型安全，性能优秀  |
| 状态同步    | gRPC     | -    | 实时性要求高     |

## 二、协议层次结构

```
┌─────────────────────────────────────────────────────────────┐
│                      应用层 (Application)                    │
│  HTTP/JSON API | TCP Binary Protocol | gRPC Protobuf       │
├─────────────────────────────────────────────────────────────┤
│                      传输层 (Transport)                     │
│  HTTP/1.1 | HTTP/3 (QUIC) | TCP | TLS                     │
├─────────────────────────────────────────────────────────────┤
│                      网络层 (Network)                       │
│  IPv4/IPv6 | TCP | UDP | QUIC                             │
└─────────────────────────────────────────────────────────────┘
```

## 四、TCP/QUIC 二进制协议

### 4.1 帧格式

```
┌─────────────────────────────────────────────────────────────┐
│  Byte 0-1          │  Byte 2-3           │  Byte 4+        │
│  MSG_ID (2B)       │  DATA_LEN (2B)     │  PAYLOAD (N)    │
│  Big-Endian        │  Big-Endian        │  JSON           │
└─────────────────────────────────────────────────────────────┘
```

- **MSG_ID**: 消息 ID，2 字节大端序
- **DATA_LEN**: 数据长度，2 字节大端序
- **PAYLOAD**: JSON 格式的请求或响应数据

## 六、ChatLoginTicket 协议

### 6.1 票据格式

```
<payload_base64url>.<signature_base64url>
```

### 6.2 Payload 结构

```json
{
  "token_type": "chat",
  "uid": 12345,
  "exp": 1711180800,，，
  "iat": 1711177200,
  "server_name": "chatserver1",
  "device_id": "device-uuid-123"
}
```

### 6.3 签名算法

- 算法：HMAC-SHA256
- Windows：使用 BCrypt
- Linux：使用 OpenSSL
- 编码：Base64URL（无填充）

### 6.4 票据验证流程

1. 解析票据字符串，提取 payload 和 signature
2. Base64URL 解码 payload 和 signature
3. 使用服务端密钥计算 HMAC-SHA256
4. 比较计算结果与传入 signature
5. 检查过期时间 (exp)
6. 验证目标服务器名称 (server_name)

## 七、传输层协议

### 7.1 TCP 协议

- 连接类型：持久连接
- 心跳间隔：30 秒
- 心跳超时：90 秒
- 最大重连次数：3
- 重连间隔：1s, 2s, 4s

### 7.2 QUIC 协议

- ALPN：`memochat-chat`
- 连接超时：5 秒
- 流控：使用 QUIC 内置流控
- 0-RTT：支持（首次连接除外）

### 7.3 HTTP/3 协议

- ALPN：`h3`
- 端口：8443
- 请求超时：30 秒
- 连接复用：支持

## 八、错误处理机制

### 8.1 自动重试


| 错误类型  | 重试次数 | 重试间隔       |
| ----- | ---- | ---------- |
| 网络断开  | 3    | 1s, 2s, 4s |
| 服务器错误 | 2    | 1s, 2s     |
| 超时    | 1    | 1s         |


### 8.2 降级策略

1. QUIC 失败 → 自动降级到 TCP
2. TCP 失败 → 自动降级到 HTTP
3. 主 ChatServer 不可用 → 自动切换到备用 ChatServer

### 8.3 离线消息处理

1. 用户离线时，消息存储到 PostgreSQL
2. 通过 RabbitMQ 队列 `chat.notify.offline` 通知
3. 用户上线后，主动拉取离线消息

## 九、安全考虑

### 9.1 传输安全

- 所有 HTTP 请求必须使用 HTTPS（生产环境）
- TCP 连接支持 TLS 加密
- QUIC 默认使用加密

### 9.2 认证安全

- Login Ticket 包含过期时间
- 每个设备有唯一 device_id
- 支持多设备同时在线（可配置）

### 9.3 消息安全

- 敏感消息支持端到端加密（可选）
- 消息签名防止篡改
- 防重放攻击（时间戳验证）

## 十一、详细错误处理流程

### 11.1 网络错误处理

```cpp
// 错误分类与重试策略

enum class NetworkError {
    // 可重试错误
    TIMEOUT,              // 超时，重试 3 次
    CONNECTION_RESET,     // 连接被重置，重试 3 次
    SERVICE_UNAVAILABLE,   // 服务不可用，重试 3 次
    RATE_LIMIT,           // 限流，退避后重试
    
    // 不可重试错误
    INVALID_REQUEST,       // 请求参数错误，不重试
    AUTH_FAILED,          // 认证失败，不重试
    PERMISSION_DENIED,    // 权限不足，不重试
    NOT_FOUND,            // 资源不存在，不重试
    
    // 需要用户介入
    SESSION_EXPIRED,      // 会话过期，需要重新登录
    SERVER_ERROR,         // 服务器内部错误，需要用户介入
    NETWORK_UNREACHABLE   // 网络不可达，需要检查网络
};

// 重试退避策略
int GetRetryDelayMs(int retry_count, NetworkError error) {
    if (error == NetworkError::RATE_LIMIT) {
        // 限流: 指数退避，最大 60 秒
        return std::min(1000 * std::pow(2, retry_count), 60000);
    } else {
        // 普通错误: 1s, 2s, 4s
        return 1000 * std::pow(2, retry_count);
    }
}
```

### 11.3 连接恢复策略

```cpp
// 断线重连策略

class ReconnectStrategy {
private:
    int _max_retries = 5;
    int _base_delay_ms = 1000;
    int _max_delay_ms = 30000;
    
public:
    // 计算重连延迟 (指数退避 + 抖动)
    int GetReconnectDelay(int retry_count) {
        int exponential_delay = _base_delay_ms * std::pow(2, retry_count);
        int capped_delay = std::min(exponential_delay, _max_delay_ms);
        
        // 添加抖动 (0.5x ~ 1.5x)
        double jitter = 0.5 + (rand() % 100) / 100.0;
        return static_cast<int>(capped_delay * jitter);
    }
    
    // 判断是否继续重连
    bool ShouldRetry(int retry_count, NetworkError error) {
        if (retry_count >= _max_retries) return false;
        
        // 认证失败不重连
        if (error == NetworkError::AUTH_FAILED) return false;
        
        // 用户主动断开不重连
        if (error == NetworkError::USER_DISCONNECT) return false;
        
        return true;
    }
    
    // 重连成功后重置状态
    void OnReconnected() {
        LOG_INFO("Reconnected successfully");
        
        // 1. 重新认证
        ReAuthenticate();
        
        // 2. 同步离线期间的消息
        SyncOfflineMessages();
        
        // 3. 恢复会话状态
        RestoreSessionState();
    }
};
```
