## 1. 项目一句话介绍

MemoChat 是一个基于 **Qt/QML 桌面客户端 + C++ 微服务后端** 的即时通讯系统，支持注册登录、好友关系、私聊、群聊、消息历史、离线消息、媒体上传、音视频通话、AI 助手、运营平台与本地部署。后端采用多服务架构，通过 **HTTP/JSON、TCP 自定义协议、QUIC、gRPC、Kafka/Redpanda、RabbitMQ、Redis、PostgreSQL、MongoDB、MinIO** 等组件实现低延迟通信、状态路由、关系存储、消息持久化和异步任务处理。

面试时可以简化成：

> 我做的是一个完整 IM 系统，不只是聊天 UI。客户端用 Qt/QML，后端用 C++ 拆成 GateServer、ChatServer、StatusServer、VarifyServer、AIServer 等服务。登录注册走 HTTP，聊天长连接走 TCP/QUIC，服务间走 gRPC，用户关系和群组存在 PostgreSQL，消息历史存在 MongoDB，在线状态和分布式路由存在 Redis，异步事件用 Kafka/Redpanda，可靠任务用 RabbitMQ。项目里还做了本地缓存、离线消息、消息状态、群聊、媒体上传、音视频和可观测性。

## 2. 简历项目描述模板

### 2.1 简短版

**MemoChat 即时通讯系统**：基于 Qt/QML + C++ 微服务实现的桌面 IM 项目，支持登录注册、好友关系、私聊群聊、离线消息、消息历史、媒体上传、音视频通话和 AI 助手。后端使用 Boost.Asio/TCP、QUIC、gRPC、Redis、PostgreSQL、MongoDB、Kafka/Redpanda、RabbitMQ 构建高并发消息链路，并通过结构化日志、链路追踪和本地部署脚本提升可观测性与可维护性。

### 2.2 详细版

- 负责设计并实现基于 **Qt/QML + C++** 的桌面 IM 客户端，使用 `AppController` 统一管理登录、聊天、联系人、群组、设置、媒体上传和音视频状态。
- 后端拆分为 `GateServer`、`ChatServer`、`StatusServer`、`VarifyServer`、`AIServer` 等服务，客户端登录注册通过 HTTP 接入，聊天通过 TCP/QUIC 长连接接入，服务间通过 gRPC 通信。
- 使用 **Redis** 存储用户 token、在线路由、ChatServer 登录数、会话映射和分布式锁；使用 **PostgreSQL** 存储用户、好友、群组、会话元数据；使用 **MongoDB** 存储聊天历史；使用 **RabbitMQ** 处理通知重试和离线任务；使用 **Kafka/Redpanda** 承载异步事件总线。
- 设计消息 ID、会话状态、好友申请、群组管理、离线消息推送、消息已读/撤回/编辑等业务链路，并做了本地 SQLite 缓存和 UI 增量刷新。
- 引入结构化 JSON 日志、trace/request/session 上下文、健康检查和部署脚本，方便本地多服务启动、定位问题和演示。

## 4. 技术栈与选型理由

### 4.1 为什么客户端用 Qt/QML？

- QML 适合快速构建现代桌面 UI，声明式布局、动画和状态绑定效率高。
- C++ 适合实现网络、缓存、模型和性能敏感逻辑。
- 项目通过 `QAbstractListModel` 暴露好友列表、会话列表、消息列表、申请列表，使 QML 只负责展示和交互。
- 与 Web 客户端相比，桌面客户端更容易集成本地文件选择、SQLite 缓存、窗口控制、托盘和系统能力。

常见追问：

**问：QML 和 C++ 怎么通信？**

答：通过 `QObject`、`Q_PROPERTY`、`Q_INVOKABLE`、信号槽和 `QAbstractListModel`。比如 QML 登录按钮调用 `controller.login(email, password)`，C++ 里 `AppController` 调用 `SessionCoordinator/AuthController` 发 HTTP 请求；聊天消息到达后 `ChatMessageDispatcher` 发信号，`AppController` 更新 `ChatMessageModel`，QML 自动刷新。

**问：为什么不用纯 QML 管业务？**

答：网络协议、缓存、消息分发和状态一致性复杂，放在 C++ 更可控；QML 负责 UI，C++ 负责状态机和业务模型，分层更清晰。

### 4.2 为什么后端用 C++？

- IM 系统对长连接、高并发、低延迟和资源占用敏感，C++ 在性能和内存控制上有优势。
- Boost.Asio/gRPC/msquic 等生态能支撑 TCP、gRPC、QUIC 多协议。
- C++ 可以在一个服务里精细控制线程模型、连接生命周期、消息队列、内存对象和异常边界。

**问：C++ 写业务服务复杂，怎么保证可维护？**

答：项目按领域拆服务和类：`ChatSessionService` 管登录/心跳/关系 bootstrap，`ChatRelationService` 管好友关系，`PrivateMessageService` 管私聊，`GroupMessageService` 管群聊，`MessageDeliveryService` 管投递，`PostgresDao/MongoDao/RedisMgr` 管存储访问。再通过结构化日志和 trace context 降低排障成本。

### 4.3 为什么使用 PostgreSQL + MongoDB + Redis？

| 存储 | 存什么 | 理由 |
|---|---|---|
| PostgreSQL | 用户、好友、群组、申请、会话元数据、权限、草稿、置顶、免打扰 | 强一致关系模型、事务、复杂查询 |
| MongoDB | 私聊/群聊消息历史、可能扩展的消息结构 | 高写入、文档灵活、消息字段可能变化 |
| Redis | token、在线状态、用户所在 ChatServer、登录计数、缓存、锁 | 低延迟、适合临时状态和路由 |
| MinIO/S3 | 头像、图片、文件、媒体资源 | 对象存储适合大文件，不压数据库 |

**问：为什么消息不用 PostgreSQL？**

答：可以用，但 IM 消息是高频追加写，消息结构可能扩展，比如回复、转发、编辑、文件、媒体元信息。MongoDB 的文档模型对消息体更灵活；PostgreSQL 继续承担关系和事务数据。面试中也可以承认：如果要做强事务、复杂审计或 SQL 分析，也可以把消息索引/摘要放 PostgreSQL，正文放 MongoDB 或对象存储。

**问：Redis 里存在线状态有什么坑？**

答：进程崩溃可能留下脏路由。项目通过登录时重置当前 ChatServer 在线集合、心跳刷新 `USER_SESSION_PREFIX` / `USERIPPREFIX` / server online set，并在投递时发现 stale route 会清理和 fallback。生产上还要加 TTL、心跳续期、节点租约和定期 reconcile。

## 5. 核心业务链路

### 5.1 登录链路

```text
客户端输入邮箱密码
  │
  ▼
GateServer /user_login
  │ 校验用户、密码、token
  │
  ├── 查询 StatusServer 获取推荐 ChatServer
  │
  └── 返回 uid、token、login_ticket、chat_endpoints
          │
          ▼
客户端根据 endpoint 连接 ChatServer TCP/QUIC
          │
          ▼
发送 MSG_CHAT_LOGIN，protocol_version=3，携带 login_ticket
          │
          ▼
ChatServer 验证 ticket，绑定 session、写 Redis 在线路由
          │
          ▼
客户端进入聊天主界面，拉取关系、会话、群组、申请
```

关键设计：

- HTTP 登录只做身份认证和分配聊天节点。
- ChatServer 登录使用 `login_ticket`，避免直接信任客户端传入的 uid/token。
- `login_ticket` 带 `uid/user_id/name/icon/target_server/expire_at_ms` 等 claims，并通过 HMAC 验证。
- ChatServer 登录后写入 Redis：用户在哪个 ChatServer、session id、server online set。

高频问题：

**问：为什么 HTTP 登录之后还要 ChatServer 登录？**

答：HTTP 登录只建立业务登录态，但聊天长连接是另一条连接，需要在 ChatServer 上绑定 socket/session 和 uid，才能投递消息、心跳和踢旧连接。分两步可以让 GateServer 专注鉴权，ChatServer 专注长连接。

**问：为什么 protocol v3 用 login_ticket，不继续用 uid/token？**

答：uid/token 由客户端直接传入容易被篡改；ticket 由服务端签发，有过期时间、目标 ChatServer 和用户资料，ChatServer 验签后才信任。这样能减少 Redis token 查询，也能避免跨节点误连。

**问：重复登录怎么处理？**

答：ChatServer 登录时根据 Redis 查询该 uid 是否已在线。如果在本节点，就找到旧 session，发送离线通知并清理；如果在其他节点，就通过 gRPC 通知对方节点踢下线。然后把 Redis 路由更新到新节点。

### 5.2 私聊消息链路

```text
A 客户端
  │ TCP/QUIC 自定义协议 ID_TEXT_CHAT_MSG_REQ
  ▼
A 所在 ChatServer
  │ 生成 msg_id / server_msg_id
  │ 写 MongoDB 消息历史
  │ 更新 PostgreSQL dialog runtime / outbox
  │ 判断 B 在线路由
  ├── B 在本节点：直接 session->Send
  ├── B 在其他节点：gRPC 通知目标 ChatServer
  └── B 离线：写离线任务/未读，登录后补推
```

关键点：

- TCP 包头通常是 `msg_id(2 bytes) + payload_len(2 bytes) + JSON payload`。
- 业务消息用 JSON，便于调试和扩展；服务间 RPC 用 Protobuf，类型更强。
- 投递层封装在 `MessageDeliveryService`，业务层不用关心本地/远程/离线细节。
- 离线消息通过登录后 `PushOfflineMessages` 补推。

**问：消息如何保证不丢？**

答：分层保证：

1. 发送入口先持久化消息或写 outbox，再尝试投递。
2. 在线投递失败时进入 RabbitMQ retry/offline task。
3. 用户登录后根据未读/历史时间戳补拉或推送离线消息。
4. 客户端本地有 SQLite 缓存，UI 不完全依赖实时网络。

可以补充：当前项目是“尽量可靠 + 可重试”的工程实现，不是严格金融级 exactly-once。真正 exactly-once 很难，通常靠消息唯一 ID + 幂等写入 + ACK/重试 + 去重实现“效果上 exactly-once”。

**问：如何处理跨节点投递？**

答：Redis 记录 `uid -> ChatServerName`，如果目标用户不在本节点，就通过 gRPC 调目标 ChatServer。若 Redis 路由过期，则扫描 server online set 或清理 stale route，再走重试/离线。

### 5.3 群聊链路

```text
发送者 -> ChatServer
  │ 校验群成员/权限/禁言
  │ 写消息历史
  │ 获取群成员列表
  │ 按成员在线路由分组
  ├── 本节点成员：直接推送
  ├── 远程节点成员：批量 gRPC 推送
  └── 离线成员：记录未读/离线
```

**问：群聊和私聊的最大区别？**

答：群聊是 fan-out，一条消息要投递给多个成员，需要考虑成员查询、批量路由、权限校验、禁言、群公告、管理员、离线未读和大群扩散成本。私聊是点对点，路由和状态简单很多。

**问：如果群很大怎么办？**

答：可以从“写扩散”和“读扩散”取舍：

- 小群：写扩散，发送时给每个成员生成未读/投递任务，读快。
- 大群：读扩散或分片拉取，发送只写一份消息，用户进来按 seq 拉取，写压力低。
- 生产中会按群规模动态策略，比如普通群写扩散，大群/频道读扩散。

### 5.4 好友申请链路

```text
A 搜索 B user_id/uid
  │
  ▼
ChatServer 查询 PostgreSQL 用户资料
  │
  ▼
A 发起好友申请 ID_ADD_FRIEND_REQ
  │
  ├── PostgreSQL 写申请记录和标签
  ├── 清理 B 的关系 bootstrap 缓存
  ├── 如果 B 在线：MessageDeliveryService 直推通知
  └── 如果 B 离线/失败：RabbitMQ relation_notify 重试
```

**问：为什么申请通知既直推又入队？**

答：实时 IM 体验要求在线用户立刻收到，所以先走在线直推；但在线路由可能过期、网络可能失败，所以失败后进入 RabbitMQ 任务队列，保证后续重试或下次登录能看到申请列表。

### 5.5 关系 bootstrap 链路

用户登录后客户端会拉取：

- 好友列表 `friend_list`
- 申请列表 `apply_list`
- 会话列表 `dialogs`
- 群组列表 `group_list`

设计点：

- v3 登录响应不再塞大量关系数据，而是登录成功后显式拉取。
- 关系数据可以短 TTL 缓存在 Redis，减少频繁进入主界面时的 DB 压力。
- 好友申请/通过后要失效相关用户的 relation cache。

**问：为什么联系人有数据但最近会话为空？**

答：联系人列表和最近会话列表是两个模型：联系人来自关系 bootstrap，最近会话来自 dialog list。如果用户有好友但没有聊天记录，服务端可能返回空 dialogs。项目客户端做了兜底：当 dialogs 为空但本地已有联系人/群组时，用联系人/群组重建“全部”会话入口。

### 5.6 媒体上传链路

```text
客户端选择文件/头像
  │ multipart/form-data
  ▼
GateServer 媒体接口
  │
  ├── 校验 token/uid
  ├── 写 MinIO/S3
  └── 返回 media url / asset id
```

**问：为什么图片文件不直接存数据库？**

答：数据库适合结构化数据和索引，不适合大量二进制大对象。文件放对象存储，数据库只存 URL/metadata，可以降低 DB 膨胀，提高 CDN/缓存扩展能力。

### 5.7 音视频通话链路

项目支持 LiveKit 相关通话流程：

- 发起通话 HTTP API 创建 room/call。
- 通过 IM 通知对方来电。
- 双方获取 LiveKit token。
- 客户端通过 QWebEngine/内嵌 Web LiveKit 页面进入房间。

**问：为什么通话不自己实现 RTP/WebRTC？**

答：音视频实时通信涉及 NAT 穿透、编码、丢包恢复、带宽估计、回声消除、设备适配等，非常复杂。项目选用 LiveKit，把重点放在 IM 业务、通话状态、邀请通知和客户端集成上。

## 6. 网络协议设计

### 6.1 HTTP/JSON

用于：

- 注册、登录、重置密码、验证码
- 用户资料更新
- 媒体上传/下载
- 通话控制
- 健康检查

优点：简单、易调试、适合请求/响应型业务。

### 6.2 TCP 自定义协议

用于聊天长连接。

典型帧格式：

```text
+---------+-------------+----------------+
| msg_id  | payload_len | JSON payload   |
| 2 bytes | 2 bytes     | N bytes        |
+---------+-------------+----------------+
```

优点：

- 比 HTTP 更轻量。
- 长连接适合实时消息推送。
- `msg_id` 做协议分发，payload 用 JSON 易扩展。

面试追问：

**问：TCP 粘包/半包怎么处理？**

答：使用固定长度头部，先读 4 字节头，解析 `msg_id` 和 `payload_len`，再按长度读 body。Boost.Asio 使用异步 read/full read，确保 body 长度完整后再投递到逻辑队列。

**问：payload_len 为什么要限制？**

答：防止恶意客户端发送超大包导致内存爆炸。项目里有 `MAX_LENGTH` 等限制，超过长度直接断开或丢弃。

### 6.3 QUIC

用于低延迟传输和未来替代 TCP。

优点：

- 基于 UDP，内置 TLS 1.3。
- 连接迁移、0-RTT、弱网表现好。
- 避免 TCP 队头阻塞，适合实时场景。

项目中支持 QUIC/TCP fallback：如果 QUIC 凭证或网络不可用，客户端可回退 TCP。

### 6.4 gRPC/Protobuf

用于服务间通信：

- StatusServer 分配 ChatServer。
- ChatServer 跨节点踢人、通知、批量群推送。
- AI 服务内部调用。

优点：强类型、IDL 清晰、性能好、适合服务间 RPC。

## 7. 并发模型与线程模型

### 7.1 ChatServer 并发模型

核心点：

- Boost.Asio 负责 TCP 异步 IO。
- 每个连接对应 `CSession`，保存 socket、session_id、uid、发送队列、心跳时间。
- 网络线程只负责读写和简单校验，完整消息投递到 `LogicSystem` 队列。
- `LogicSystem` 有 worker 线程处理业务回调，避免在 IO 回调里做 DB/MQ 等耗时工作。
- 发送端有 send queue，避免多个 async_write 并发写同一个 socket。

**问：为什么网络 IO 和业务逻辑要分开？**

答：如果在 IO 回调里直接查数据库或执行业务，单个慢操作会阻塞事件循环，影响所有连接。拆成 IO 层和逻辑 worker 后，IO 层保持快速响应，业务层可以并行处理。

**问：如何保证 session 生命周期？**

答：异步回调中使用 `shared_from_this()` 保活，防止 socket 回调回来时对象已析构。关闭时统一 `Close()`、从 server session map 清理、从 UserMgr 清理 uid->session 映射。

### 7.2 异常边界

项目中需要重点回答：

- C++ worker 线程里未捕获异常会导致进程 terminate。
- detached thread 未捕获异常也会 terminate。
- 因此在 `LogicSystem::workerLoop` 和离线消息推送等异步线程中要加 `try/catch`，记录结构化错误日志，避免服务崩溃。

**问：你遇到过什么真实 bug？怎么排查？**

可以回答：

> 登录后 ChatServer 曾经崩溃。最开始没有日志，因为异常没有捕获。后来在逻辑回调层加了异常边界，日志显示 `bad variant access`。继续排查发现 v3 登录包只带 `login_ticket`，但服务端先读取了缺失的 `uid/token` 字段。修复方式是先读取 `protocol_version`，v3 先验证 ticket，再从 claims 取 uid；旧协议才读 uid/token。这个问题体现了协议演进时必须兼容旧字段和做好异常边界。

## 8. 数据模型与存储设计

### 8.1 用户与关系

PostgreSQL 中一般包括：

- `user`：uid、user_id、email、pwd、nick、icon、desc、sex。
- 好友关系表：owner uid、friend uid、备注、标签、状态。
- 好友申请表：from uid、to uid、状态、标签、时间。
- 群表：group_id、group_code、name、icon、owner_uid、announcement、member_limit。
- 群成员表：group_id、uid、role、mute_until、status。
- dialog meta/runtime：置顶、免打扰、草稿、最后消息、未读数。

**问：好友关系为什么通常要双向存？**

答：查询某个用户的好友列表是高频操作。双向存储可以让 `WHERE owner_uid = ?` 快速查询，不需要每次 OR 两个字段或做复杂 join。缺点是写入时要保证两条记录一致，因此需要事务或幂等补偿。

### 8.2 消息 ID

项目里可用 Snowflake 思路：

- 时间戳 + datacenter id + worker id + sequence。
- 多 ChatServer 实例必须有不同 worker id，否则会冲突。
- 本地部署脚本中会给 chatserver1-4 设置不同 WorkerId。

**问：Snowflake 有什么问题？**

答：

- 依赖机器时间，时钟回拨会导致 ID 冲突或乱序。
- worker id 必须唯一。
- 同毫秒 sequence 耗尽要等待下一毫秒。
- 解决方式包括 NTP 监控、回拨保护、Redis/DB 分配 worker id、或者使用 UUID/ULID 等替代。

### 8.3 消息历史

MongoDB 文档可以包含：

- `msg_id/server_msg_id`
- `from_uid/to_uid/group_id`
- `content/msg_type`
- `created_at`
- `reply_to_server_msg_id`
- `forward_meta`
- `edited_at_ms/deleted_at_ms`

**问：如何分页拉历史消息？**

答：不要用大 offset，而是基于 `created_at + msg_id` 做游标分页。客户端保存 before timestamp/msg_id，请求更早消息，数据库用索引范围查询，避免 offset 越大越慢。

## 9. 缓存、一致性与高可用

### 9.1 Redis 缓存内容

- 用户基础信息缓存：`USER_BASE_INFO + uid`
- token：`USERTOKENPREFIX + uid`
- 在线路由：`USERIPPREFIX + uid -> chatserverName`
- session：`USER_SESSION_PREFIX + uid -> sessionId`
- server 在线集合：`SERVER_ONLINE_USERS_PREFIX + serverName`
- relation bootstrap 缓存：`relation_bootstrap_uid`
- 登录数：`LOGIN_COUNT`

### 9.2 缓存一致性策略

- 用户资料变更后删除/刷新用户基础缓存。
- 好友申请、同意好友后删除双方 relation bootstrap 缓存。
- 在线状态通过心跳刷新，进程启动时清理本节点在线集合。
- 搜索用户时可以绕过陈旧缓存，优先读 PostgreSQL 并刷新缓存，保证头像等资料最新。

**问：缓存穿透/击穿/雪崩怎么处理？**

答：

- 穿透：空值缓存、参数校验、布隆过滤器。
- 击穿：热点 key 互斥锁/逻辑过期/后台刷新。
- 雪崩：TTL 加随机抖动、多级缓存、限流降级。

当前项目更多是本地开发和工程演示，生产可以继续加这些策略。

### 9.3 在线状态一致性

**问题**：进程崩溃时 Redis 可能还显示用户在线。

**解决思路**：

- 心跳续期 + TTL。
- ChatServer 启动时清理自己名下的 online set。
- 投递时发现本节点没有 session，则清理 stale route。
- 定时 reconcile：扫描 online set，和本地 UserMgr session map 对比。

## 10. 消息队列设计

### 10.1 Kafka/Redpanda 用在哪里？

用于事件总线：

- 私聊事件
- 群聊事件
- 关系变更事件
- 登录审计事件
- 用户资料变更事件

适合“事件广播、异步边效应、可扩展消费”。

### 10.2 RabbitMQ 用在哪里？

用于可靠任务：

- 离线通知
- 投递重试
- 好友关系通知
- outbox repair

适合“任务分发、ACK/NACK、延迟重试、死信队列”。

**问：Kafka 和 RabbitMQ 区别？为什么两个都用？**

答：Kafka/Redpanda 更适合高吞吐事件流、多个消费者订阅、按 topic 分区保留日志；RabbitMQ 更适合任务队列、路由灵活、ACK/NACK、重试和死信。IM 里聊天事件和审计适合 Kafka，可靠通知/延迟重试适合 RabbitMQ。

**问：如何保证 MQ 消息不重复消费？**

答：MQ 只能保证至少一次更常见，业务层要幂等。比如消息表用唯一 msg_id，好友申请用 `(from_uid, to_uid)` 唯一约束或 upsert，通知可以带 task_id 去重。

## 12. 安全设计

### 12.1 登录安全

- HTTP 登录校验密码，返回 token 和 chat login ticket。
- ChatServer 验证 ticket 签名和过期时间。
- ticket 可绑定目标 ChatServer，避免拿 ticket 连接错误节点。
- Redis token 用于旧协议或部分 HTTP API 鉴权。

### 12.2 常见安全问题

**问：如何防止伪造 uid 发消息？**

答：ChatServer 登录后 session 绑定 uid，后续消息不应该完全信任 payload 里的 fromuid，而应该以 session uid 为准校验。如果 payload fromuid 和 session uid 不一致，拒绝或覆盖。

**问：如何防重放？**

答：ticket 有过期时间；可以加入 nonce/jti，Redis 记录已使用 ticket；请求级别加入 timestamp + nonce + HMAC。聊天消息可用 msg_id 幂等去重。

**问：密码如何存储？**

答：生产应该使用 bcrypt/argon2/scrypt 加盐哈希，不能明文或普通 MD5。项目面试中要说明如果当前是开发实现，后续会替换为强哈希。

## 13. 可观测性与排障

### 13.1 日志

- `message`

- `session_id`
- `uid`
- `attrs`

优点：方便 grep、聚合、按 trace 查链路。

### 13.2 典型排障案例

#### 案例 1：ChatServer 登录崩溃

现象：客户端登录失败，ChatServer 退出。  
定位：加异常边界后日志出现 `logic.callback.exception` 和 `bad variant access`。  
原因：v3 登录包没有 uid/token，服务端先读取缺失字段。  
修复：先读 protocol_version，v3 验 ticket 后从 claims 取 uid，旧协议才读 uid/token。

#### 案例 2：好友申请对方收不到

现象：A 发申请，B 在线但收不到。  
原因：通知只依赖 RabbitMQ 异步任务，如果任务总线异常或消费失败，实时通知缺失。  
修复：好友申请和同意申请先用 `MessageDeliveryService` 在线直推，失败再发 RabbitMQ 重试。

#### 案例 3：联系人有但“全部”会话为空

现象：联系人 tab 有好友，但全部会话列表为空。  
原因：全部绑定 dialogModel，dialog list 返回空时清空了 dialogModel。  
修复：当 dialogs 为空但本地已有联系人/群组时，用本地关系数据重建会话入口。

#### 案例 4：头像显示默认图

现象：搜索到用户但头像是默认图。  
原因：用户资料 Redis 缓存过期或旧值。  
修复：按 UID 搜索优先读 PostgreSQL 最新资料并刷新缓存；资料变更要清理缓存。

## 14. 性能优化点

### 14.1 服务端

- IO 线程和业务 worker 分离。
- 长连接异步读写，避免线程阻塞。
- send queue 串行化 socket 写，避免并发 async_write。
- Redis 缓存在线路由和用户资料，降低 DB 查询。
- 群推送按目标 ChatServer 分组批量 gRPC。
- 历史消息用游标分页，避免 offset 慢查询。
- 异步事件和任务用 MQ 解耦主链路。

### 14.2 客户端

- QML 列表使用 model 增量 upsert，避免全量刷新。
- 消息本地缓存减少重复网络拉取。
- 登录后并行 bootstrap dialogs/contacts/groups/apply。
- 图片和头像使用 URL，避免大对象进模型。
- `ListView` 设置 cacheBuffer、懒加载历史消息。

**问：如何压测？关注哪些指标？**

答：

- 连接数：ChatServer 可维持多少 TCP/QUIC 长连接。
- 登录 QPS：GateServer + StatusServer + Redis 压力。
- 消息 QPS：私聊/群聊发送、投递、持久化吞吐。
- 延迟：端到端 P50/P95/P99。
- 资源：CPU、内存、FD/socket、DB 连接池、Redis QPS、MQ backlog。
- 错误：投递失败率、重试数、断线率、慢查询。

## 15. 架构优点与不足

### 15.1 优点

- 微服务拆分清晰，登录、状态、聊天、验证、AI 分工明确。
- 多协议支持，HTTP 负责短请求，TCP/QUIC 负责长连接，gRPC 负责服务间调用。
- 多存储按场景选型，PostgreSQL/MongoDB/Redis/MinIO 各司其职。
- MQ 解耦异步任务和实时主链路，提高可靠性和扩展性。
- 客户端状态模型较完整，支持联系人、会话、群组、申请、消息缓存。
- 有结构化日志、部署脚本、本地多实例启动和一定的可观测能力。

### 15.2 不足与改进

- 本地登录密码缓存目前偏开发便利，生产应接入系统安全凭据或只保存 token。
- 消息可靠性还可以继续强化，例如严格 ACK、幂等去重、消息补偿任务。
- Redis 在线状态需要 TTL/租约/reconcile 进一步完善。
- 大群架构需要根据规模做读扩散/写扩散切换。
- 用户密码存储、接口限流、防重放、防刷验证码等安全细节可继续增强。
- 可以引入统一配置中心、服务注册发现和灰度发布。
- 可以将部分手写 JSON 协议迁移到 Protobuf 或 FlatBuffers，减少字段兼容问题。
- QUIC 凭证和 fallback 策略可进一步完善。

## 16. 高频面试问答总表

### 16.1 项目宏观

**Q1：这个项目最核心的难点是什么？**

A：核心难点是实时 IM 的端到端链路：用户登录后要分配 ChatServer，长连接要绑定用户和在线路由，消息要在本节点/跨节点/离线三种情况下正确投递，还要持久化、未读、重试、历史拉取和客户端 UI 增量刷新。难点不是单个接口，而是多服务、多协议、多存储之间的一致性和异常处理。

**Q2：你负责了哪些部分？**

A：可以按实际情况回答：客户端 QML/C++ 状态管理、登录/聊天/联系人/群组模块；ChatServer 登录、关系、投递、离线消息；Redis 在线状态；PostgreSQL/MongoDB DAO；本地部署和日志排障等。

**Q3：如果让你重新设计，会改什么？**

A：会把安全凭据、协议 IDL、在线状态 TTL、消息 ACK 和幂等去重进一步标准化；引入服务发现和配置中心；对大群采用读扩散；对客户端登录密码缓存改用系统 Credential Store。

### 16.2 C++/并发

**Q4：Boost.Asio 异步模型怎么工作？**

A：注册 async_read/async_write 回调，io_context 事件循环等待内核 IO 完成后调用回调。为了避免对象提前析构，session 在回调中捕获 shared_ptr。耗时业务不在 IO 回调里做，而是投递到逻辑队列。

**Q5：如何避免多个线程同时写 socket？**

A：每个 session 有发送队列和锁，只有队列为空时启动 async_write；写完成回调里 pop 当前包，再继续写下一个，保证同一 socket 写操作串行。

**Q6：worker 线程异常怎么办？**

A：worker loop 外层或每个 callback 调用处必须 catch，否则 C++ 线程异常会 terminate 进程。catch 后记录 `msg_id/error/trace_id/session_id`，必要时给客户端返回错误。

### 16.3 网络协议

**Q7：为什么不用 WebSocket？**

A：WebSocket 是可行方案，易穿透、生态好。但项目选择自定义 TCP/QUIC 是为了学习和控制协议细节，减少 HTTP upgrade 额外层，同时支持更灵活的 msg_id 二进制头和 QUIC fallback。生产可以根据场景选择 WebSocket 或 TCP/QUIC。

**Q8：TCP 如何处理断线重连？**

A：客户端检测连接断开或心跳超时后重连 Gate/Chat；服务端心跳维护 session；重连后重新 ChatServer 登录，更新 Redis 路由；历史/未读通过 bootstrap 或离线消息补齐。

### 16.4 数据库

**Q9：PostgreSQL 连接池怎么设计？**

A：维护固定数量连接，业务线程获取连接，使用后归还；如果连接异常则重建；避免每次请求新建连接造成高开销。池大小要结合 worker 数、DB max connections 和慢查询情况调优。

**Q10：MongoDB 消息表怎么建索引？**

A：私聊可按 `(owner_uid/peer_uid, created_at, msg_id)` 或会话维度索引；群聊按 `(group_id, seq/created_at)`。历史分页使用游标，避免 offset。

### 16.5 分布式与一致性

**Q11：如何保证好友申请不丢？**

A：先写 PostgreSQL 申请记录，这是最终可恢复的数据；在线用户直推失败后入 RabbitMQ 重试；用户登录时 relation bootstrap 会从 DB 拉取申请列表，所以即使实时通知丢了，也能在下次登录看到。

**Q12：如何处理消息重复？**

A：消息带唯一 msg_id/server_msg_id，DB 写入使用唯一键或 upsert，客户端按 msg_id 去重；MQ 重试导致重复投递时，客户端和服务端都要幂等。

**Q13：如何做水平扩展？**

A：ChatServer 多实例，每个实例管理一部分长连接；StatusServer 根据 Redis 登录数/节点状态分配；跨节点通过 Redis 路由 + gRPC；DB/MQ/Redis 也可以集群化。

### 16.6 客户端

**Q14：Qt Model 如何避免全量刷新卡顿？**

A：对新增或更新项使用 upsert 和 dataChanged；必要时批量更新；历史消息分页加载；图片懒加载；避免每条消息都触发昂贵布局。

**Q15：登录账号缓存怎么实现？安全吗？**

A：当前用 QML Settings 保存最近账号列表，登录页启动时读取最近一次账号自动填充，右侧下拉可选择历史账号。这个功能偏本地开发便利；生产中应使用系统凭据管理器或只保存 refresh token，不明文保存密码。

### 16.7 运维和部署

**Q16：本地怎么启动？**

A：先用 `tools\scripts\status\deploy_services.bat` 把 build 产物部署到 `infra/Memo_ops/runtime/services`，再用 `tools\scripts\status\start-all-services.bat` 启动 MinIO、VarifyServer、GateServer、StatusServer、4 个 ChatServer 等服务；停止时使用同目录下的 `stop-all-services.bat`。

**Q17：怎么定位线上问题？**

A：先看结构化日志的 `trace_id/request_id/session_id/uid`，确认链路在哪一层失败；再看 Redis 在线路由、DB 数据、MQ backlog、服务端口和进程状态；最后用客户端日志和服务端 event 交叉验证。

## 18. 可以主动展示的亮点

- **协议演进**：从 uid/token 登录演进到 `login_ticket`，提高安全性和跨节点准确性。

- **存储分层**：关系/消息/缓存/对象存储明确分工。

## 19. 面试官可能深挖的“坑点”与推荐回答

### 19.1 明文密码缓存

**坑点**：你刚加了账号密码缓存，面试官可能问安全性。

推荐回答：

> 这个功能是为了本地演示方便，用 QML Settings 保存最近登录账号密码。严格来说生产不应该明文存密码。正式方案我会改成只保存账号和 refresh token，密码交给系统 Credential Store，比如 Windows Credential Manager/DPAPI、macOS Keychain，并提供“记住密码”开关。

### 19.2 JSON 字段缺失导致异常

**坑点**：C++ JSON variant 访问可能抛异常。

推荐回答：

> 协议字段要做兼容读取，不能直接 `root["field"].asInt()` 假设字段存在。我后来改成 `glaze_safe_get` 或先判断 `isMember`，特别是协议版本演进时，v2/v3 字段不同，要先读版本再分支解析。

### 19.3 exactly-once

**坑点**：面试官问消息是否 exactly-once。

推荐回答：

> 分布式 IM 很难真正 exactly-once。更实际的是 at-least-once + 幂等，实现“效果上 exactly-once”。也就是消息有唯一 ID，DB 唯一键，客户端去重，MQ 重试允许重复但不产生重复业务效果。

### 19.4 Redis 在线状态不一致

推荐回答：

> Redis 在线路由是临时状态，必须允许不准。投递时如果 Redis 指向本节点但本地没有 session，就清理 stale route；心跳续期；节点启动清理自己的 online set；生产再加 TTL 和定时 reconcile。

### 19.5 大群扩散

推荐回答：

> 当前适合中小群写扩散。大群需要分层：消息只写一次，在线成员按节点批量推，离线成员用 seq 拉取，避免给每个离线成员写大量任务。
