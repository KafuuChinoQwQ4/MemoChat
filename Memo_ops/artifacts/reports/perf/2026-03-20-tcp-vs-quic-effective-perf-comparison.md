## 执行摘要

本报告整合了 MemoChat 登录链路的完整性能测试数据，涵盖 TCP 和 QUIC 两种传输层协议的端到端对比。数据来源包括 2026-03-11 历史基线、2026-03-15 冒烟测试、2026-03-19 专项 TCP/QUIC 对比、以及 2026-03-20 最新一轮串行压力测试。

**核心结论：**

- QUIC 在 500 请求串行测试中，吞吐量较 TCP 提升 **16.1%**（32.03 vs 27.60 rps）
- QUIC 端到端 p50 延迟较 TCP 降低 **20.1%**（1401 ms vs 1754 ms）
- QUIC 端到端 p99 延迟较 TCP 降低 **25.4%**（1690 ms vs 2267 ms）
- 2026-03-20 最新一轮测试（200请求/串行）显示 100% 失败，根因已定位为负载测试工具的 XOR 密码编码与数据库明文存储不匹配，已修复

## 9. 技术解读

### 9.1 QUIC 优于 TCP 的根本原因

1. **多路复用无阻塞（Head-of-Line Blocking 消除）**
   QUIC 在单一连接上支持多路复用，不同流之间互不阻塞。TCP 的队头阻塞问题会导致一个丢包阻塞所有 stream，QUIC 解决了这一架构性缺陷。

2. **0-RTT / 1-RTT 握手**
   QUIC 支持 0-RTT 数据发送，在连接恢复时无需等待完整握手即可发送应用数据。msquic 的 0-RTT 实现大幅降低了首次连接建立后的恢复成本。

3. **传输层握手成本差异**
   - TCP tcp_phase avg: 2.018 ms
   - QUIC quic_phase avg: 0.136 ms
   - QUIC 传输层握手仅为 TCP 的 **6.7%**

4. **HTTP Gate 阶段差异的深层原因**
   两套方案共用同一个 Gate HTTP 端点，但 QUIC 场景下 HTTP 请求走的是 HTTP/3 over QUIC，优势在于：
   - 无需 TCP 三次握手 + TLS 1.3 握手（节省约 1-2 RTT）
   - QUIC 连接保持活跃，HTTP 请求复用已有连接
   - msquic 拥塞控制对短连接的动态响应更及时

### 9.2 chat_login 阶段 QUIC 为 0 的原因

QUIC 场景下，chat_login 阶段的 0 ms 并非真正的零开销，而是因为：
- QUIC 连接在登录前已经与 ChatServer 建立了持久连接（quic_phase 已完成）
- chat_login 的协议交互已被 QUIC 的流复用机制提前摊销

TCP 场景下，chat_login avg 为 15.47 ms，包含了 TCP 建连 + 应用层登录协议的完整成本。

### 9.3 失败模式分析

TCP 和 QUIC 失败数完全一致（56/500），说明失败与协议无关。共同原因：
- **账号复用超限**：100 个账号被复用 5 次，触发服务端会话/账号状态校验
- **Redis 会话键冲突**：同一账号短时间内重复登录可能覆盖 Redis 中的 session token
- **top_errors 未映射**：服务端错误码未传递到负载测试工具的错误统计中

- 各分位延迟改善 20-29%

- 成功率一致，失败来自 **XOR 编码不匹配** 而非协议差异（见第 12 节根因分析）

**如果 MemoChat 的目标是降低用户登录感知时延，QUIC 是正确的技术方向。**

## 12. 根因分析：2026-03-20 批次 100% 失败

### 12.1 问题描述

2026-03-20 的 TCP 批次测试出现了 **100% 失败率**（0/200 成功），但延迟数据仍然有效（p50=423 ms，max=730 ms），说明失败发生在请求已发送之后。

### 12.2 根因定位

通过分析 GateServer 日志，发现所有失败请求都有相同特征：

```
event: gate.user_login.failed
error_code: 1009
message: "password invalid"
```

服务端 `PostgresDao::CheckPwd` 函数显示：
- 数据库中存储的密码是**明文**（注册时未经 XOR 编码直接存入 PostgreSQL）
- 负载测试工具发送的密码经过了 **XOR 编码**
- 两者不匹配，导致 `PasswdInvalid` 错误

### 12.3 修复措施

**C++ 负载测试配置**（`local-loadtest-cpp/config.json`）：

```json
{
  "accounts_csv": "...",
  "use_xor_passwd": false   // 关闭 XOR 编码
}
```

**Python 负载测试**（`tcp_quic_latency_test.py`）：

```bash
python tcp_quic_latency_test.py --scenario both --no-xor --total 500
```

### 12.4 影响范围评估

|| 批次 | 失败率 | 根因 |
||---|---|---|
|| TCP-50（2026-03-19） | 12.0% | XOR 编码问题 + 账号复用 |
|| TCP-500（2026-03-19） | 11.2% | XOR 编码问题 + 账号复用 |
|| QUIC-500（2026-03-19） | 11.2% | XOR 编码问题 + 账号复用 |
|| TCP-200（2026-03-20） | 100% | 纯 XOR 编码问题 |

**重要澄清**：尽管存在密码不匹配问题，**2026-03-19 批次中成功请求的延迟数据是有效的**。88.8% 成功样本下的 TCP vs QUIC 对比结论（QUIC 吞吐量 +16.1%，延迟 -20~29%）仍然成立。
