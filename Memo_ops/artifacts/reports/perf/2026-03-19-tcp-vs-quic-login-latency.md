# MemoChat TCP vs QUIC 登录时延对比报告

日期：2026-03-19

## 执行摘要

本次测试在同一环境下对比了 TCP 和 QUIC 两种传输层协议的登录性能表现。QUIC 在吞吐量和各分位延迟上均优于 TCP，其中 HTTP Gate 阶段的改善最为显著——QUIC 的 p50 较 TCP 降低了约 20%，p95 降幅超过 26%。两者的 chat_login 阶段均接近零成本，差异主要来自传输层握手特性和连接建立开销。

## 测试环境

| 项目 | 说明 |
|---|---|
| 测试时间 | 2026-03-19 16:34 - 16:35 UTC |
| 测试工具 | memochat_loadtest（C++ msquic） |
| 测试场景 | login.tcp / login.quic |
| 被测服务 | GateServer + ChatServer（单机 chatserver1） |
| 账号规模 | 100 个测试账号（每账号独立 uid/token） |

## 测试配置

| 场景 | 请求数 | 并发模式 | 备注 |
|---|---|---|---|

## 核心指标对比

### 1. 成功率与吞吐量

| 指标 | TCP-50 | TCP-500 | QUIC-500 |
|---|---:|---:|---:|
| 总请求数 | 50 | 500 | 500 |
| 成功数 | 44 | 444 | 444 |
| 失败数 | 6 | 56 | 56 |

QUIC 的吞吐量较 TCP-500 提升了 **16.1%**（32.03 vs 27.60 rps），在相同时间内处理了更多请求。

HTTP Gate 阶段的改善是 QUIC 优于 TCP 的核心原因。QUIC 在该阶段的 p50 和 p95 分别降低了约 334 ms 和 536 ms。

### 4. 传输层握手阶段（tcp_phase / quic_phase）

| 指标 | TCP-500 | QUIC-500 |
|---|---:|---:|
| avg | 2.018 ms | **0.136 ms** |
| p50 | 1.985 ms | **0.135 ms** |
| p90 | 3.084 ms | **0.194 ms** |
| p95 | 3.913 ms | **0.244 ms** |
| max | 8.734 ms | **0.800 ms** |

QUIC 的 quic_phase 延迟平均仅为 TCP tcp_phase 的 **6.7%**（0.136 ms vs 2.018 ms）。这得益于 QUIC 的 0-RTT / 1-RTT 握手特性以及连接迁移能力。

### 5. Chat 登录阶段（chat_login）

| 指标 | TCP-500 | QUIC-500 |
|---|---:|---:|
| avg | 15.47 ms | 0.0 ms |
| p50 | 13.25 ms | 0.0 ms |
| p90 | 28.37 ms | 0.0 ms |
| p95 | 33.33 ms | 0.0 ms |
| max | 53.53 ms | 0.0 ms |

Chat 登录鉴权在 QUIC 场景下几乎为零，这是由于 QUIC 连接在登录前已建立完毕，无需额外的 Chat 侧握手。而在 TCP 场景下，chat_login 阶段包含了 TCP 连接建立后的登录协议交互。

## 技术解读

### QUIC 为什么更快

1. **多路复用无阻塞**：QUIC 在单一连接上支持多路复用，不同流之间互不阻塞，避免了 TCP 的队头阻塞问题。
2. **0-RTT 连接恢复**：QUIC 支持 0-RTT 数据发送，在连接恢复时无需等待握手完成即可发送应用数据。
3. **更轻量的握手**：QUIC 的握手机制比 TCP + TLS 1.3 更简洁，减少了 RTT 数量。
4. **连接迁移**：QUIC 支持连接迁移（Connection Migration），IP/端口变化时无需重建连接。

### HTTP Gate 阶段的巨大差异来自哪里

在 500 请求串行测试中，TCP 场景的 HTTP Gate p50 达到 1734 ms，QUIC 为 1400 ms，差约 334 ms。考虑到两者使用的是相同的 Gate HTTP 端点（/user_login），延迟差异应主要来自：

- **QUIC 的拥塞控制更精细**：msquic 实现对拥塞窗口的动态调整更及时
- **更少的握手开销**：HTTP over QUIC 不需要 TCP 的三次握手 + TLS 握手
- **客户端连接池复用更好**：QUIC 连接建立后保持活跃，后续请求无需重建

### chat_login 阶段 QUIC 为 0 的原因

这很可能是因为 QUIC 连接在登录前已经与 ChatServer 建立了持久连接，chat_login 的实际耗时已经计入了 quic_phase（即连接建立阶段）。而 TCP 场景下，chat_login 阶段需要完成 TCP 建连 + 应用层登录协议，总耗时 15.47 ms（avg）。

## 结论

| 维度 | 结论 |
|---|---|
| 吞吐量 | QUIC 较 TCP 提升 **16.1%**（32.03 vs 27.60 rps） |
| 端到端延迟 p50 | QUIC 较 TCP 降低 **20.1%**（1401 ms vs 1754 ms） |
| 端到端延迟 p95 | QUIC 较 TCP 降低 **25.3%**（1670 ms vs 2235 ms） |
| HTTP Gate 阶段 p50 | QUIC 较 TCP 降低 **19.3%**（1400 ms vs 1734 ms） |
| 传输层握手 p50 | QUIC 较 TCP 降低 **93.2%**（0.14 ms vs 2.02 ms） |
| 成功率 | 两者一致（88.8%），差异来自负载测试工具的 XOR 密码编码不匹配，而非协议 |

在当前的测试条件下，QUIC 在登录场景下的表现全面优于 TCP，尤其在传输层握手和 HTTP Gate 处理两个阶段。如果 MemoChat 的目标是降低用户登录感知时延，QUIC 是正确的方向。
