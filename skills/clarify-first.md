# MemoChat 先澄清再执行

用于任务存在关键未知项、且这些未知项会改变实现、验收、数据、接口或风险边界时。这个 skill 的目标是先把必须决策的点问清，再开始实现，避免靠猜测推进。

## 触发条件

- 至少有一个关键决策点尚未确定，且不同答案会导向不同的文件、行为、契约、验证或回滚方案。
- 代码、文档、配置或已有实现无法直接给出单一安全路径。
- 不是单纯的实现细节补全，也不是已经有明确方向的普通任务。

## 规则

- 只要不确定会影响实现、数据、接口、UI、验证、回滚或风险边界，就先问，不要自己拍板。
- 优先一次收集当前最重要的信息；如果问题很多，按影响从高到低分批问，但不要只问一个泛泛问题。
- 问题要具体、可执行，优先覆盖：
  - 目标和非目标
  - 预期行为和验收标准
  - 必须保留的现有行为
  - 约束条件和不可变边界
  - 是否接受替代方案，以及偏好的方向
- 如果本地代码、文档、配置或已有实现能回答，就先查再问。
- 如果存在多个合理方向，先给出简短选项，再请用户选择。
- 在关键未知项未澄清前，不要开始实现、重构或大范围改写。
- 如果已经能按仓库现有规则直接推进，只是还缺少局部实现细节，不要滥用这个 skill，改走 `task.md`、`debugging.md` 或 `parallel-agents.md`。
- 如果用户已经给出明确方向，就不要为了提问而提问，直接执行。
- 对于高风险、跨模块、跨服务、数据契约、UI 交互或运行时行为变更，默认先收集更多信息。
